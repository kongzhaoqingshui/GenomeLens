# 上游修改汇总

本文档记录 GenomeLens 当前对 vendored JCVI 的**直接本地补丁**，方便后续升级上游版本时快速复刻、核对和评估。

## 当前状态

- 实际运行的 vendored 源码目录：`engines/jcvi/src/jcvi/`
- 本地参考上游目录：`references/upstream/jcvi/src/jcvi/`
- 当前上游版本基线：`1.6.6`
- 当前 patchset 标识：`windows-genomelens`

当前仓库中与 vendored JCVI 来源相关的关键 Git 节点：

- `60915aa1`：初次将 vendored JCVI 纳入 GenomeLens 仓库
- `126ab6c5`：将 vendored JCVI 从 `1.6.5` 升级到 `1.6.6`

## 作者与来源说明

GenomeLens 仓库保留的是**实际导入与维护历史**，不会通过重写 Git author 的方式伪装成上游作者提交。

上游作者信息通过以下方式保留：

- `engines/jcvi/licenses/JCVI-LICENSE.txt`
- `references/upstream/jcvi/` 中的上游项目元数据
- `engines/jcvi/vendor/upstream-jcvi/SOURCE.md`
- `jcvi_genomelens probe` 输出中的 `jcvi_upstream_version` 与 `patchset`

## 当前 direct patch 列表

下列内容才算“相对上游的本地补丁”。`engines/jcvi/src/jcvi_genomelens/` 中的 GenomeLens 自研 engine 包装代码不算上游补丁。

### 1. `jcvi/_version.py`

- 文件：`engines/jcvi/src/jcvi/_version.py`
- 用途：固定 vendored 版本元数据，便于打包、探测和摘要输出保持一致
- 当前值：`1.6.6`

### 2. `jcvi/apps/base.py`

- 文件：`engines/jcvi/src/jcvi/apps/base.py`
- 用途：补齐 Windows 运行兼容性
- 当前本地补丁点：
  - `SIGPIPE` 访问加保护，避免 Windows Python 缺失该符号时报错
  - `which()` 允许补全 `.exe` 候选，便于解析 `blastn`、`makeblastdb` 等可执行文件
  - shell 调用不再在 Windows 下强依赖 `/bin/bash`

### 3. `jcvi/graphics/tree.py`

- 文件：`engines/jcvi/src/jcvi/graphics/tree.py`
- 用途：兼容当前 GenomeLens Windows 依赖环境
- 当前本地补丁点：
  - 优先使用 `ete4`
  - 若不可用则回退到 `ete3`

## 升级上游时的处理规则

1. 先同步新的上游参考树到 `references/upstream/jcvi/`
2. 再更新 `engines/jcvi/src/jcvi/`
3. 对照本文档和 `engines/jcvi/vendor/upstream-jcvi/patchset.md` 复刻仍然需要保留的本地补丁
4. 更新：
   - `engines/jcvi/src/jcvi/_version.py`
   - `engines/jcvi/src/jcvi_genomelens/_version.py`
   - `engines/jcvi/vendor/upstream-jcvi/SOURCE.md`
   - `engines/jcvi/vendor/upstream-jcvi/patchset.md`
   - `engines/jcvi/vendor/upstream-jcvi/hashes.txt`

## 建议的后续提交策略

以后每次升级 vendored JCVI，尽量拆成两类提交：

1. `chore(vendor): sync upstream JCVI x.y.z snapshot`
2. `patch(jcvi-vendor): apply GenomeLens patchset`

这样可以把“上游同步”与“GenomeLens 本地补丁”清楚分开，也更适合后续审计和答辩。
