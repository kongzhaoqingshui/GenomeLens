# 代码整洁改进建议（已归档）

> **归档说明**：本文档为历史参考。其中所列清理项（死代码删除、目录迁移、`run_summary` 去重、方法专属字段下沉、CLI 工作台调整、旧兼容垫片移除等）**大部分已实施**；文中 `python/`、`jcvi-genomelens/` 等旧路径已分别迁移到 `platform/` 与 `engines/jcvi/`，阅读时请以当前目录为准。

---

> **历史文档提示**：本文写于仓库布局迁移之前，文中的 `python/`、`jcvi-genomelens/` 等路径已分别迁移到 `platform/` 与 `engines/jcvi/`。具体执行时应以当前实际目录为准，思路仍可参考。

本文基于对 GenomeLens 源码的一次通读，记录发现的"脏"点，并给出让项目更干净的改进路径。按影响面从大到小排列，每条都标注了证据、风险与建议动作。

适用范围：本文只针对项目自有代码与仓库结构，不评判 vendored 的 JCVI 上游源码本身（那是第三方代码，不应改动）。

> 阅读须知：本文把发现分成三类，请勿一律按"删除"处理——
> - **真该清理的**：误提交进 Git 跟踪的本地参考、有活替代品的并存死代码。
> - **架构预留、暂勿动的**：尚未接线、但路线图文档明确要用的占位组件。
> - **该顺架构方向收口、而非就地去重的**：当前重复其实是平台核心协议的雏形，应朝统一 schema 收口。

## 一、`references/upstream/` 误进 Git 跟踪（最高优先级，但不是"删源码"）

### 现状

仓库被 Git 跟踪的文件共 658 个，其中 `references/` 占 331 个。其中 `references/upstream/` 有 **317 个文件被 Git 跟踪**，主要是：

- `references/upstream/jcvi/`：上游 JCVI 的完整镜像（含 `.github/`、`docker/`、`tests/` 等），315 文件级别。
- `references/upstream/haiant_plugin/`：含 `phylo_tree.zip` 与 `开发手册.md`。

关键事实核实：

- `.gitignore` **完全没有提到 `references/upstream/`**，但该目录却被纳入了 Git 跟踪。
- 抽样 diff 显示这份 JCVI 镜像与 `jcvi-genomelens/src/jcvi/`（真正 vendored、打包运行用）只有 2 个文件不同（`apps/base.py`、`graphics/tree.py`）。这说明它的用途是**本地对照参考**——因为 vendored JCVI 会被本地改动，需要留一份上游原样来比对改了哪里。
- 全仓没有任何代码 `import` 它，纯静态镜像。

### 定性

这**不是**"该删的重复源码"，而是"**该被 Git 忽略却误提交的本地参考**"。它在磁盘上有存在价值（对照 vendored 的本地补丁），问题只在于它不该进版本库——和 `toolchains/`、`references/downloads/` 同类，README 和开发手册都已说明这类生成物/离线参考不进 Git 跟踪，唯独 `references/upstream/` 漏配了忽略规则。

### 问题

- 仓库体积和 clone 时间被第三方镜像主导，自有代码（python 83 文件）被淹没。
- code review、搜索、IDE 索引都要扛着这几百个无关文件。
- 镜像一旦进库还会随提交漂移，失去"上游原样"的参考意义。

### 建议（保留磁盘文件，停止跟踪）

1. 在 `.gitignore` 增加忽略规则，例如：
   ```gitignore
   # 本地上游对照参考，不进 Git 跟踪
   references/upstream/**
   ```
   如需保留目录占位，可配 `!references/upstream/.gitkeep`。
2. 用 `git rm -r --cached references/upstream/` 把它从索引移除（**不加 `--cached` 以外的删除**，磁盘文件原样保留）。
3. `references/upstream/haiant_plugin/`（含 zip 与开发手册）大概率同理，一并纳入忽略。
4. 给真正 vendored 的 `jcvi-genomelens/src/jcvi/` 补一份 `VENDORED.md`：记录上游版本号/commit、以及本地打了哪两个补丁（即上面 diff 出的两个文件），让"上游对照"这件事有据可查，而不是靠仓库里压一整棵镜像树。

> 注意：上一版本文档把这条写成"删除 `references/upstream/jcvi/`"，定性有误。正确动作是**停止跟踪、保留本地**，不是物理删除。

## 二、并存死代码（可删）与架构预留（勿动），需区分对待

通读发现几个文件在 `python/` 与 `integrations/` 全量搜索下没有任何引用，但它们的性质**不同**，处理方式也不同。

### 2.1 可删：有活替代品的并存死代码

| 文件 | 状态 | 证据 |
| --- | --- | --- |
| `python/src/genomelens/models.py` | 无人 import | 全仓搜索 `genomelens.models` / `from genomelens import models` 零命中；实际使用的是 `engine/core/models.py` |
| `python/src/genomelens/errors.py` | 无人 import | 同上，`genomelens.errors` 零命中；实际使用的是 `app/errors/exceptions.py` |

这两个文件里定义的 `DiagnosticMessage` / `PathPair` / `NamedResource` 等，都有**正在被使用的同名/同职责替代版本**。根级这两个文件是早期分层时留下的空壳，并存会让人误以为它们是公共入口。

`parsers.py` 仍被 `request_normalizer.py` 使用（`parse_formats`），属于活代码，**保留**。

建议：
1. 删除 `models.py`、`errors.py`（删前用 `git grep` 再确认一次，含字符串形式的动态引用）。
2. 如果 `errors.py` 的初衷是给外部调用方提供稳定的错误导出入口，那就把它写成真正的公开 API（在 `__init__.py` 里 re-export 并加测试），否则删掉。二选一，不要留半成品。

### 2.2 勿动：有路线图依据的架构预留

| 文件 | 状态 | 性质 |
| --- | --- | --- |
| `python/src/genomelens/data/cache/sqlite_index.py` | 当前零引用 | 路线图明确要用的 fingerprint 缓存预留 |

核实结论——这个文件**不能**当死代码删：

- docstring 自述"预留给后续 fingerprint 复用"。
- `docs/CLI.md` 已存在 `clean [--cache]` 命令形态，指向一个缓存层。
- `docs/开发手册/架构调整/最终架构目标.md` 的 Artifact Layer 反复强调 `provenance(来源追踪)`、可追踪结果资产，与 fingerprint 缓存属于同一蓝图。

它与 2.1 的本质区别：2.1 是"有活替代品的重复空壳"，本条是"尚未接线但路线图已规划的未来组件"。

建议：
1. **暂不删除**。在文件 docstring 或开发手册里补一句它对应路线图的哪一层（Artifact Layer / fingerprint 缓存），并加上"尚未接线"标注。
2. 若团队确认短期内（如本轮改版计划范围内）不会接入，再考虑移除并用 issue 记录设想——但这是需要团队确认路线图后的决定，不是清理动作可以顺手做的。

## 三、`workflow_controller.py` 的 run_summary 重复：朝统一 schema 收口，而非就地抽函数

### 现状

`app/controller/workflow_controller.py`（381 行，自有代码里最大的文件）中，`_run_pairwise_mcscan` 和 `_run_multi_species_mcscan` 各自手写了一个结构高度相似的 `run_summary` 字典：

- 两处都重复拼装 `status` / `schema_version` / `workflow` / `jcvi_backend` / `final_figures` / `artifact_index` / `logs` / `ui` / `scoring` 等字段。
- `ui` 块和 `scoring` 块在两个方法里几乎逐字重复。
- `scoring` 的 message "机器学习评分模块尚未接入当前工作流。" 在两处硬编码。

### 区分两种"重复"

核实架构文档后，这里要明确**哪些是该消除的重复，哪些是有意的演进结构**：

- **该消除的**：上述 `ui` / `scoring` / 公共头部字段的逐字重复，以及硬编码中文 message。这部分是单纯的拷贝。
- **不该当重复消除的**：`_run_multi_species_mcscan` 把任务拆成多个 pairwise 再聚合，这个结构**本身不是重复**——它正是 `最终架构目标.md` 里 `species[]` 模型 + "全组合两两比较"工作流的当前落地形态，是有意的演进结构，不要为了"看起来不重复"去合并它。

### 更重要的方向：这是平台协议的雏形

`run_summary` 的字段（`artifact_id` / `produced_by` / `preview` / `artifact_index` 等）正是两份架构文档的核心目标：

- `最终架构目标.md` 第 4 节 Artifact Layer 要把这些标准化为统一的 `manifest/result/summary` 协议与 artifact 元数据。
- `分析命令改版计划.md` 要把结果统一收口为 `AnalysisResult`，并由 `AnalysisDispatcher` 统一处理 summary。

因此，**就地抽一个本地 `build_run_summary()` helper 虽然能立即去重，但很可能是临时方案**——平台核心一旦落地，这个 helper 还要为统一 schema 重写。

### 建议

1. **短期**：可以先抽出 `ui` / `scoring` / 公共头部的构造（哪怕是个本地 helper + 把占位 message 提为模块级常量），消除逐字重复、避免改 schema 时漏改一处。这是低风险的即时收益。
2. **但要标注定位**：在该 helper 或文档里注明"这是 Artifact/Summary schema 的预备收口点，最终应随 `分析命令改版计划.md` 的 `AnalysisResult` / Artifact Layer 统一"，避免后人误以为这就是终态。
3. **真正的收口**应按改版计划的阶段推进（统一 `AnalysisResult`、dispatcher 统一产出 summary），而不是在 controller 里堆本地工具函数。
4. 顺带可把 `_run_pairwise_mcscan` 200+ 行的长方法按"准备输入 / 定位工具链 / 运行 engine / 组装 summary"拆成更小的私有方法，降低单方法认知负担——这一条与架构方向无冲突，可独立做。

## 四、CLI 直接用 print，缺少统一输出层

### 现状

`cli/commands/` 下的 `analyze.py`、`check.py`、`clean.py`、`config.py`、`main.py` 大量直接调用 `print(...)`（含 `print(..., file=sys.stderr)`、`print("\n" * 40)` 这种清屏技巧）。

### 问题

- 输出散落在各命令里，无法统一控制（比如加 `--quiet`、`--json`、颜色开关、重定向到日志）。
- `print("\n" * 40)` 这类靠打印空行"清屏"的写法脆弱且依赖终端。
- stdout（数据/JSON）和 stderr（人读信息）的边界没有统一约定。

### 建议

1. 已有 `cli/ui.py`（236 行，含 `render_*` 系列函数），把所有面向用户的输出统一收敛到一个 writer/console 抽象里，命令内部不再直接 `print`。
2. 明确约定：机器可读输出（JSON）走 stdout，人读提示走 stderr，方便管道使用。
3. 清屏改用成熟方式（如 `rich` 的 console，或至少封装成一个 `clear_screen()` 函数）。

## 五、`except Exception` 的使用（已合理，无需整改）

### 现状

自有代码里有 5 处 `except Exception`（engine 侧另有 3 处），都加了 `# noqa: BLE001` 并写了理由注释，例如：

- `workflow_controller.py:293`：多物种汇总需要记录单个 pairwise 子任务失败原因。
- `cli/main.py:121`：CLI 边界统一映射预期与非预期错误。
- `toolchain_installer.py:180/202`：安装命令需上报任意下载/解压失败。

### 评价

这些位置（CLI 最外层、子任务聚合、外部命令边界）用宽捕获是**合理**的，而且都带了 noqa 理由，比裸 `except:` 干净得多。这一条不算需要整改的"脏点"，列在这里是为了说明：宽捕获已被有意识地控制在边界层，继续保持这个纪律即可，不要让它向核心逻辑蔓延。

## 优先级与建议执行顺序

1. **让 `references/upstream/` 停止 Git 跟踪**（`.gitignore` 加规则 + `git rm -r --cached`，保留磁盘文件）。一次性把数百个误跟踪文件移出版本库，收益最大、风险最低。给 vendored JCVI 补 `VENDORED.md`。
2. **删除并存死代码** `models.py` / `errors.py`（删前再 grep 确认）。
3. **保留 `sqlite_index.py`**，仅补充"架构预留、尚未接线"的标注；是否最终移除需团队确认路线图，不在本轮清理范围内。
4. **消除 run_summary 的逐字重复**（抽 helper + 常量），并标注其为 Artifact/Summary schema 的预备收口点；真正统一随 `分析命令改版计划.md` 推进。
5. **统一 CLI 输出层**，消除散落的 `print`。
6. 收尾：开发手册补一段 `*models.py` 职责说明。

第 1、2 步是纯配置/删除，做完仓库会明显变干净，且几乎零回归风险（建议各自一个独立 commit，配合 `python -m pytest platform/tests` 验证）。第 4、5 步是渐进式重构，可分多次进行，且需对齐架构文档的方向。
