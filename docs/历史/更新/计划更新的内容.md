### v0.9.x — 补齐 v1.0.0 核心能力

主题：在 v1.0.0 正式发布前，以 0.9.x patch 形式补齐核心流程能力。

- [ ] **v0.9.1：细致的 `run.log`（2）**

  - 统一日志格式，支持 `--verbose` 与 `--log-level`

- [ ] **v0.9.2：外部配置与智然体插件（3、6）**
  - 恢复 `analyze run <request.json>` 入口，复用 `AnalysisRequest` 模型与调度器
  - 提供 JSON schema 与示例
  - 智然体插件仅暴露一站式 JCVI 出图流程，输出稳定插件契约
  - **合并顺序**：先合并 `analyze run JSON`，再合并智然体插件
- [ ] **v0.9.3：JCVI 子任务命令接口（4）**
  - 将引擎侧子任务（如 `graphics_synteny`、`graphics_dotplot`、`local_synteny`）映射为 CLI 子命令
  - 支持 `analyze mcscan jcvi <subtask> <input> <output>` 形式的自动流程
  - 保持现有 `analyze mcscan jcvi` 一站式入口不变
- [ ] **v0.9.4：多物种局部共线性图与出图优化（1、5）**
  - ≥3 物种时聚合参考物种与所有目标物种的局部共线性结果，输出多物种局部共线性总图
  - 出图自动优化（可选开关）：最终图片尺寸、轨道链接重写（`0→1`、`0→2` 改为 `0→1→2`）、blocks 跨染色体切除
  - 每项优化独立参数控制，默认关闭

## 建议开发顺序

1. **先落地 2（run.log）**：属于基础设施，后续所有功能都能受益。
2. **并行推进 3（analyze run）与 6（智然体插件）**：都涉及外部接口契约，可共用请求模型抽象。
3. **再推进 4（JCVI 子任务 CLI）**：为 1 和 5 提供更细粒度的调试与调用入口。
4. **接着做 1 和 5**：依赖 4 的子命令能力，也依赖现有引擎出图逻辑。
5. **最后启动 7 和 8**：GUI 和 macOS 属于独立大模块，可在主线稳定后并行分支开发。

### 并行原则

1. **v0.9.1（run.log）建议先开第一个 PR**：
   - run.log 会贯穿所有 runner，先合并可以为其他人的调试提供便利。
   - 其它分支只需在合并后补一行 `log.task_started/finished` 即可，冲突面小。

2. **v0.9.2 的两项（analyze run JSON + 智然体插件）建议串行合并，但可并行开发**：
   - analyze run JSON 改动集中在 platform shell 侧，定义 `AnalysisRequest` schema 与 `analyze run <request.json>` CLI。
   - 智然体插件改动集中在 `integrations/` 侧，目前插件已经会写 `genomelens_request.json`，只是把 CLI 调用从 `analyze mcscan` 改成 `analyze run`。
   - **合并顺序**：先合并 `analyze run JSON`，再合并智然体插件。插件分支在 analyze run 合并后 rebase 并切换调用方式。
   - 如果希望完全并行，可先开一个极小的 PR 只定义/锁定 `AnalysisRequest` JSON schema，两个分支都基于该 schema 开发。

3. **v0.9.3（JCVI 子任务 CLI）可与 v0.9.2 并行启动**：
   - 只要不动 `analyze mcscan jcvi` 一站式入口的默认行为，就不会影响 B/C。
   - 但建议在 v0.9.2 合并后再合入 v0.9.3，这样插件和 JSON 入口都能自然兼容新的子命令。

4. **v0.9.4 的两项建议等 v0.9.3 合并后再大规模开发**：
   - 多物种局部共线性图和出图优化都可能在 CLI 层增加新子命令或参数。
   - 等 v0.9.3 把子命令框架确定后，再基于该框架扩展，避免重复调整 CLI 结构。
   - 但调研、引擎侧预研、测试数据准备可以先做。

### 代码审查重点

- run.log：是否覆盖所有子任务，日志格式是否统一。
- analyze run JSON：JSON schema 是否与 CLI 参数等价，错误提示是否清晰。
- 智然体插件：是否按 `analyze run <request.json>` 调用，插件契约是否稳定，是否只依赖公开接口。
- JCVI 子任务 CLI：是否破坏现有 `analyze mcscan jcvi` 默认行为。
- 多物种局部共线性图：是否复用现有 `local_synteny`，是否引入新的 ID 冲突。
- 出图优化：每项优化是否独立开关，默认是否关闭。

## 风险与注意事项

- **5 的优化开关**：跨染色体切除若默认开启，可能改变老用户输出；必须显式参数控制。
- **4 的子任务接口**：需避免与现有 `--jcvi-workflow` 参数语义冲突，建议子命令优先、参数回退。
- **8 的 macOS 移植**：工具链路径、动态库、字体渲染与 Windows 差异较大，建议单独 `mac` 分支长期维护。
- **并行冲突热点**：`cli/commands/analyze.py`、`core/models.py`、`app/controller/runners/_shared.py` 是高频冲突区，建议改动前在协作群中同步。
