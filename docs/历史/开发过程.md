# GenomeLens 开发过程

> 本文是开发回忆录，用于记录 GenomeLens 从选题探索、原型验证、架构重整到当前阶段的历程。文中可能保留旧命令、旧入口和阶段性表述，不代表当前 CLI 的最新状态；当前 CLI 请以 `docs/CLI.md` 和 `docs/使用方法/README.md` 为准。

本文用于记录 GenomeLens 从选题探索、原型验证、架构重整到当前阶段的开发过程，便于后续汇报、复盘和交接。全文按“我们”的共同开发历程展开。

## 一、方向探索与原型验证阶段

这一阶段发生在正式进入 Agent 协助开发之前，是 GenomeLens 从选题想法走向工程项目的起点，也是项目开发史的第一阶段。我们在这一阶段完成了方向筛选、领域理解、JCVI 试跑、Windows 兼容探索和初始架构搭建，也正是在这一阶段认识到后续必须进行工程化重整。

### 1.1 选题背景

最开始，我们面对的是毕业设计选题本身。这个问题看似只是“做什么项目”，实际上牵涉到几个更具体的判断：

- 需要和生物信息学方向保持紧密关联。
- 需要有足够的技术深度，不能只停留在简单脚本。
- 需要体现完整的软件实现过程，而不是只做一次数据处理。
- 需要贴近真实科研场景，最终能解释它为什么有实际价值。
- 需要在毕业设计周期内形成可演示、可汇报、可继续扩展的闭环。

在早期筛选中，我们看过多个可能方向，例如序列处理、可视化小工具、数据格式转换、注释辅助、系统发育分析、组学数据整理等。这些方向都能做，但问题也很明显：有些过于零散，有些更像单个脚本，有些难以体现工程结构，有些则很难自然扩展成一个长期项目。

### 1.2 方向收敛

经过反复比较，我们逐渐把注意力放到了比较基因组学。这个方向吸引我们的地方在于，它不是单纯的数据清洗，也不是只画一张图，而是天然包含了一条完整链路：

- 数据准备。
- 格式转换。
- 外部算法调用。
- 工具链依赖管理。
- 结果解释。
- 图形输出。
- 报告整理。

这意味着比较基因组学既有明确的生物学意义，也有足够的软件工程空间。对于一个希望做成真实工具的毕业设计项目来说，它比单点脚本更适合作为核心方向。

### 1.3 共线性分析理解

在进一步了解比较基因组学的过程中，我们接触到了 synteny(共线性分析)。最初很多概念并不直观，例如：

- ortholog(直系同源)。
- paralog(旁系同源)。
- anchors(锚点)。
- blocks(区块)。
- dotplot(点图)。
- synteny(共线性图)。
- karyotype(核型图)。

这些词一开始只是文献、教程和命令说明里的术语，后来才逐渐和实际文件、分析过程、图件结果对应起来。我们逐渐意识到，共线性分析不是单纯“画一条线”，而是提取不同基因组之间的保守结构关系，并用图形和结构化结果辅助解释物种演化、同源区域、候选基因和染色体片段关系。

这个理解过程直接改变了项目定位。如果没有真正理解共线性分析，项目很容易被做成“上传两个文件、调用一个命令、下载一张图”。而在理解其科研用途后，我们更明确地意识到：GenomeLens 应该降低共线性分析门槛，把输入准备、工具链依赖、参数选择、运行过程、结果整理和图形生成组织成一个完整流程。

### 1.4 JCVI 选型

在寻找实现路径时，我们接触到了 JCVI。JCVI 已经提供了比较成熟的比较基因组学能力，尤其是 MCScan、catalog、dotplot、synteny、karyotype 等模块，基本覆盖了我们想做的核心流程。

因此，早期技术路线逐渐明确：

- 不从零实现共线性算法。
- 以 JCVI 作为核心分析和绘图能力来源。
- 用 Python 构建外层工具，负责输入、配置、运行、输出和交付。
- 面向 Windows 使用场景做额外封装。
- 后续再考虑插件、图形界面和更复杂的分析能力。

这个判断让项目从“想做一个生物信息学工具”变得具体起来：我们要做的是围绕 JCVI 的 GenomeLens，而不是另一个孤立脚本。

### 1.5 Windows 兼容探索

真正开始实操后，第一个现实难点很快出现：JCVI 对 Windows 并不友好。JCVI 本身更偏向类 Unix 环境，直接在 Windows 上运行时，依赖环境、命令调用、路径处理、外部工具链、图形输出和打包运行都会暴露问题。

这一阶段我们进行了大量探索：

- 反复尝试不同 Python 环境，确认哪些依赖能在 Windows 下安装和运行。
- 直接运行 JCVI 命令，观察它在哪些路径、文件名或外部命令上失败。
- 构造最小 BED+CDS 样例，排除输入数据复杂度带来的干扰。
- 尝试 GFF+FASTA 输入，理解预处理和格式转换需求。
- 定位 BLAST+、ImageMagick 等外部工具，判断它们应该由系统提供、项目提供还是打包时处理。
- 观察 JCVI 生成的 anchors、simple、blocks、layout、seqids 等中间文件，理解它们在流程中的位置。
- 尝试把一次手动命令过程整理成可重复执行的脚本或入口。

这些尝试看起来琐碎，但它们让我们真正摸到了项目的难点。共线性分析工具不是只要会调用库就能完成，还必须处理环境、路径、外部程序、输入格式、输出归档和错误解释。

### 1.6 初始架构形成

随着问题不断被解决，我们逐渐搭出了施工计划中的初始架构。那时的架构已经包含一些后来仍然保留下来的想法：

- 外层 Python 程序。
- JCVI 相关代码。
- 工具链目录。
- 示例数据。
- 文档目录。
- 打包脚本。
- 插件适配雏形。

这版架构不是完全没有设计意识。相反，它是我们在不断解决实际问题时自然长出来的第一版骨架。它证明了项目方向有现实基础，也让我们第一次看到了从输入文件到共线性结果的路径。

但这版架构也有明显特点：它更多是问题驱动，而不是架构驱动。哪里报错，就在哪里补一段处理；哪里路径不对，就在那里做一次转换；哪里命令缺参数，就直接把参数拼上；哪里输出找不到，就临时增加一次文件扫描。这样推进很快，但项目开始形成越来越多隐性的依赖关系。

### 1.7 原型结果

在这一阶段，我们确实实现过一个“能跑”的效果：

- 能够在一定条件下把输入文件送入 JCVI。
- 能够跑出共线性分析相关结果。
- 能够看到图形产物。
- 能够初步理解 JCVI 输出目录和中间文件。
- 能够证明 Windows 兼容并非完全不可行。

这个结果非常重要，因为它证明项目方向不是空想。JCVI 的能力可以被调动起来，GenomeLens 作为工具也有现实基础。

但这个“能跑”没有带来真正的踏实感。只要换一个目录、换一台机器、换一组输入、换一种启动方式，问题就可能重新出现。某些流程依赖手动准备，某些路径写法不统一，某些错误只在很后面才暴露，某些结果文件虽然生成了，但没有被清楚登记和解释。

### 1.8 架构问题暴露

随着原型推进，我们开始意识到早期架构过于脆弱。主要问题包括：

- CLI、JCVI 调用、Windows 兼容、工具链、输入预处理、输出整理和插件适配混在一起。
- 旧实现更多依赖临时拼接命令，而不是稳定协议。
- 代码中既有正式逻辑，也有兼容补丁、临时调试和实验性脚本。
- 文档和实现没有形成稳定对应关系。
- 很多能力只能解释“为什么当时这样写”，但很难解释“以后为什么还应该这样写”。
- 如果继续堆功能，后续多物种、GUI、插件、跨平台都会越来越难接入。

这个阶段让我们第一次强烈体会到软件工程思维的重要性。以前我们更关注“这个功能能不能实现”，后来开始关注：

- 这个功能应该放在哪一层。
- 这个接口以后会不会被 GUI 或插件复用。
- 这个路径是不是只适合当前机器。
- 这个错误能不能被清楚报告。
- 这个结果能不能被审计。
- 这个文档是不是和真实实现一致。

这种思维变化，是项目进入工程化阶段的前提。

### 1.9 第一次架构重整准备

在架构问题逐渐暴露后，我们开始准备第一次重要重整。重整的核心不是“把代码写得更漂亮”，而是重新划定项目边界：

- Python 外壳应该负责输入、配置、工具链、工作区、调用和摘要。
- JCVI engine 应该负责真实 JCVI 分析与绘图。
- 插件应该只做平台参数适配，不应该绕过主流程。
- 工具链和 Windows 兼容应该被封装，而不是散落在业务逻辑里。
- 输入请求、engine 清单和运行摘要应该成为稳定协议。
- 已完成能力、诊断能力和未来预留能力必须分清楚。

施工计划就是在这一背景下逐渐成形的。它不是凭空写出来的，而是早期踩坑之后的一次系统整理。它把散落的经验、问题、目标和重构方向组织起来，让后续开发不再只是“今天修一个报错，明天补一个参数”，而是有了阶段目标和验收口径。

### 1.10 阶段转折

在完成方向探索、原型验证和第一次架构反思后，我们开始学习和理解 Agent 等现代化开发方式。这个变化并不是简单地把一部分代码交给工具生成，而是重新理解复杂项目的推进方式。

对于 GenomeLens 这样的项目，Agent 协助开发的价值在于：

- 可以持续阅读和维护跨目录上下文。
- 可以同时检查代码、测试和文档的一致性。
- 可以协助进行大范围重构。
- 可以帮助补齐测试和 CI。
- 可以把架构计划转化为可执行任务。
- 可以推动项目从可运行原型走向可维护版本。

因此，后续开发不是从零开始，而是建立在这一阶段的前史之上：

- 我们已经完成毕业设计方向筛选，确认比较基因组学适合工具化。
- 我们已经理解 synteny(共线性分析)的基本意义。
- 我们已经确认 JCVI 是合适的核心能力来源。
- 我们已经在 Windows 上实操过 JCVI，知道兼容难点集中在哪里。
- 我们已经搭出过初始架构，并用它验证过方向可行。
- 我们也已经认识到早期架构不能支撑多物种、插件、GUI、跨平台和未来平台化目标。

这一阶段是 GenomeLens 开发过程中的分水岭。转折点之前，我们主要在回答“这个方向能不能做出来”；转折点之后，我们开始回答“怎样把它做成长期可维护、可交付、可扩展的工程”。

## 二、工程化阶段的项目目标

我们最初面对的是一个目标清晰但工程化程度还需要补齐的比较基因组学项目。项目的长期目标并不是只封装一个单次 JCVI 命令，而是建设一个面向实际科研工作的 GenomeLens 平台：

- 从 2 个到任意多个物种的 GFF、FASTA、BED、CDS 等原始或预处理文件出发。
- 在正式分析前完成参数预设、配置读取、工具链检查和输入校验。
- 全程调用真实的 JCVI 分析与绘图能力，而不是用简化算法冒充正式结果。
- 最终生成可审计、可复现、可发表前继续美化的共线性分析结果。
- 未来逐步扩展为比较基因组学平台，继续接入更多算法、深度学习能力和 AI Agent 工作流。

围绕这个目标，我们从一开始就确定了几个基本原则：

- Python 侧作为交互入口和工作流外壳，负责 CLI、配置、输入、输出、工具链、日志、结果汇总。
- `jcvi-genomelens` 作为独立 JCVI engine(引擎)，负责真实 JCVI 调用、共线性分析和图形生成。
- HAIant 插件作为平台适配层，不直接实现算法，只做参数翻译和运行时调用。
- 各层之间尽量通过 manifest(JSON 清单)、request(JSON 请求)、summary(JSON 摘要) 通信，降低耦合。
- Windows first(Windows 优先)，但架构上逐步为 macOS、Linux 和未来 GUI 留出空间。

## 三、施工计划落地与仓库治理

开发初期，我们先读取施工计划，确认环境、依赖和参考资料的位置。项目环境被统一修正为 `genomelens` conda 环境，Python 解释器版本固定为 3.12。JCVI 源码仓库、智然体插件开发资料和示例代码都放在 `references/upstream` 下，作为对照和迁移参考。

早期为了避免意外丢失进度，我们采用过全量备份策略，将项目定期备份到同盘 `mydata` 目录。后来项目连接 GitHub 后，我们将主要风险控制方式切换为 Git 管理和远端 master 分支，减少本地重复备份带来的混乱。

在仓库治理上，我们做过几轮清理和规范化：

- 将施工计划打包为 `施工计划.zip`，并加入忽略规则，避免施工资料误混入代码版本。
- 删除阶段性待办文档，将真正需要保留的内容转入正式文档或计划文档。
- 为 `.gitignore` 增加中文注释，明确各类忽略项的用途。
- 清理无用旧文件、占位代码、重复上游源码和不再适合新架构的入口。
- 对上游 JCVI 的本地改动建立汇总文档，方便未来对接新版本时检查差异。

这些动作让仓库逐渐从“能跑的施工现场”收敛为“可以继续长期维护的项目”。

## 四、三层结构逐步成型

我们在开发过程中逐渐稳定了 GenomeLens 的三层结构：

```text
platform/                入口侧 shell(外壳) / 平台核心雏形
engines/jcvi/            独立 JCVI engine(引擎)
integrations/            平台适配与插件
```

`platform/` 负责交互入口。它不直接导入上游 `jcvi`，而是负责把输入整理成结构化请求，再调用 engine。这个边界非常重要，因为它让 CLI、图形界面、插件、Agent 和未来其他引擎可以共用同一套分析协议。

`engines/jcvi/` 负责 JCVI 能力的真实接入。我们将上游 JCVI 放入可控的 vendored(内置上游) 目录，并针对 Windows 运行做必要兼容。engine 通过 `probe` 暴露能力，通过 `run --manifest` 执行具体工作流。

`integrations/haiant_plugin/` 负责智然体平台适配。它不直接实现共线性分析，也不直接引用 GenomeLens 内部 Python 包，而是把平台参数转换为 GenomeLens 可理解的请求，再调用打包后的 runtime(运行时)。

这个结构让项目从早期“命令脚本集合”逐渐变成了有边界、有协议、有交付路径的应用型科研工具。

## 五、JCVI 主流程接入

项目最核心的问题是：是否真正接入了 JCVI 的共线性分析流程。我们围绕这个问题做了多轮检查和实现。

首先，我们确认不能只停留在“能调用部分命令”或“能生成示例图”的程度，而要做到从输入文件到最终图件的完整链路：

- 读取 BED+CDS 或 GFF+FASTA。
- 对 GFF+FASTA 做预处理，生成 JCVI 需要的 BED 和 CDS。
- 定位 BLAST+、JCVI engine 等外部工具。
- 生成 engine manifest(JSON 清单)。
- 调用 JCVI catalog 主流程，执行真实比较分析。
- 生成 anchors、simple、blocks 等中间文件。
- 调用 JCVI graphics 能力生成 dotplot(点图)、synteny(共线性图)、karyotype(核型图) 等图形输出。
- 汇总 engine summary(引擎摘要) 和 shell run summary(运行摘要)。

随后我们检查 `probe` 能力声明和 dispatcher(调度器) 实际支持能力是否一致。早期存在“声明了 dotplot、karyotype，但 dispatcher 没有完整工作流”的风险。我们将能力声明改为分层：

- `capabilities` 只声明真实可调度 workflow(工作流)。
- `bundled_jcvi_modules` 说明 vendored JCVI 中存在的模块。
- 避免上层误以为某个 JCVI 模块已经被完整产品化接入。

在能力补齐阶段，我们接入和规范了这些 workflow：

- `mcscan_pairwise`：基础双物种 JCVI 共线性分析。
- `graphics_synteny`：默认主链路，除 synteny 图外同时生成 dotplot。
- `graphics_dotplot`：独立点图工作流。
- `graphics_karyotype`：核型图工作流，可在缺少 layout/seqids 时生成默认布局。
- `catalog_ortholog`：保留正交目录分析入口。

我们明确了一个底线：正式结果必须来自真实 BLAST+ 和 JCVI，不启用简化共线性 fallback(降级算法)。当 `allow_simplified_fallback=true` 时，流程会给出清晰错误，避免把非正式结果误认为可发表结果。

## 六、从双物种到基础多物种流程

项目目标要求支持 2 到 n 个物种。我们先把双物种真实链路跑通，再基于双物种流程扩展多物种编排。

当前多物种流程采用的是 all-vs-all pairwise(全组合两两比较) 策略：

- 传入 2 个物种时，直接运行双物种 JCVI 主流程。
- 传入 3 个及以上物种时，自动拆分为多个 pairwise 子任务。
- 每个 pairwise 子任务复用双物种真实流程。
- 汇总所有子任务的结果、图件、日志和状态。

这种做法的好处是稳：我们没有重新发明一套未经验证的多物种算法，而是把已经真实可运行的双物种链路作为基本积木，先完成基础多物种分析能力。

目前已经完成的是基础多物种 JCVI 共线性分析编排。尚未完成的是跨全部物种的一张全局美化总图、全局 layout/seqids 自动优化、全局排序策略、候选评分策略和更复杂的多物种可视化布局。

这个边界在文档中也被反复同步，避免把“基础多物种 pairwise 编排”描述成“完整多物种最终美化总图”。

## 七、配置系统收口

早期配置模型中已经存在 `blastn_path`、`makeblastdb_path`、`magick_path`、默认线程数、默认 block size、默认输出格式等字段，但真实流程没有完全使用这些配置。我们将配置系统逐步收口，让配置真正进入运行链路。

配置优先级被固定为：

```text
CLI 显式参数 > config 文件 > 环境变量 > 系统 PATH > 打包资源 > 本地 toolchains
```

随后我们将配置分化为两类：

- GenomeLens 主配置：工具本身的运行设置、工作区设置、通用默认值。
- JCVI 配置：JCVI engine、BLAST+、ImageMagick、workflow、绘图等与子工具相关的设置。

`analyze` 和 `check` 支持 `--config`、`--jcvi-config`，也支持 `GENOMELENS_CONFIG`、`GENOMELENS_JCVI_CONFIG` 环境变量。这样既方便普通场景使用默认配置，也方便高级场景、CI 和打包环境显式指定路径。

## 八、工具链策略与 Windows CI

GenomeLens 依赖外部工具链，尤其是 BLAST+ 和可选 ImageMagick。我们围绕这部分处理了多个实际问题。

早期工具链逻辑偏向“项目内直接调用”。后来我们改成更合理的顺序：

- 先检查系统内是否已有可用工具。
- 如果系统工具不存在，再尝试使用项目本地 toolchains。
- 如果本地也没有，再按需要下载。
- 下载和安装过程尽量写入缓存，避免 CI 每次重复下载。

在 Windows CI 中，我们遇到过 BLAST+ 安装器目录扫描问题：GitHub runner 上归档结构与预期不完全一致，安装器直接遍历不存在的 `bin` 目录导致失败。我们修复了安装器，使它同时支持两种结构：

- 可执行文件直接在候选目录下。
- 可执行文件位于候选目录的 `bin/` 下。

随后我们优化了 CI：

- 合并重复 Windows workflow。
- 保留一套快速稳定的必跑 Windows CI。
- 加入 BLAST+ cache(缓存)。
- cache 命中后仍做轻量检查，确认 `blastn.exe` 和 `makeblastdb.exe` 存在。
- 保留手动 release smoke(发布冒烟测试)，用于更完整的构建、打包、插件和检查流程。

我们讨论过“是否应该把外部工具调用链加入 CI”。最终判断是：真实 BLAST+ 和 JCVI 是 GenomeLens 的核心风险点，应该保留在必跑 CI 中，但要通过缓存、轻量样例和手动 smoke 控制成本。

## 九、Windows 兼容与上游改动管理

项目是 Windows first，因此我们对上游 JCVI 做了必要兼容。我们没有把这些改动散落成不可追踪的状态，而是补充了上游修改汇总，记录哪些地方为了 Windows、打包或 GenomeLens engine 做了调整。

我们还讨论了跨平台策略。起初设想过 Windows、macOS 分支分开演进；后来更倾向于“单一核心，多平台适配”的方式：

- 核心 Python 包和 engine 协议保持单一。
- 平台差异封装在路径定位、工具链安装、打包脚本、启动脚本和 GUI 层。
- Windows 继续优先交付。
- macOS、Linux 作为后续平台适配，不复制整套业务逻辑。

针对 Cython，我们也确定了方向：如果性能收益重要，且未来会继续开发 Cython 模块，那么应保留 Cython 能力，但要将其作为核心性能层的一部分进行规范管理。Windows first 不意味着排斥跨平台，而是先保证 Windows wheel/build 可控，再逐步补齐 macOS 和 Linux 的构建矩阵。

## 十、未来平台架构与 Tauri GUI

随着项目定位从“JCVI 封装工具”扩展到“比较基因组学平台”，我们重新梳理了未来架构：

- GenomeLens Core：统一核心协议、任务模型、配置、工具链、数据模型和结果模型。
- Engine Layer：JCVI、未来其他比较基因组学工具、深度学习模型、AI Agent 工具链。
- Workflow Layer：从输入识别、参数推荐、任务编排到结果汇总。
- Interface Layer：CLI、Tauri GUI、HAIant 插件、未来 Web/Agent 接口。
- Runtime Layer：打包、工具链定位、平台适配、日志、缓存和安全策略。

在 GUI 方向上，我们明确倾向使用 Tauri 做 Windows/macOS GUI 层。Tauri 不直接承载算法，而是作为界面层，通过统一 request/summary 协议调用核心能力。这样 CLI、GUI 和插件都不会形成三套不同逻辑。

我们为这个方向写了几份架构文档：

- 最终架构目标。
- 温和架构过渡。
- 多平台兼容方案。

后来这些文档被精修并打包到“架构调整”目录下，形成更清晰的长期演进材料。

## 十一、机器学习与 AI Agent 能力预留

我们讨论过 GenomeLens 未来不只局限于 JCVI，而是会加入机器学习和 AI Agent。围绕共线性分析场景，已经明确可以放入的机器学习能力包括：

- 候选优先级排序。
- 同源可信度评分。
- 启动子差异风险评分。
- 区段综合评分。

这些能力不应该直接塞进 JCVI engine，也不应该污染 CLI 命令解析。更合适的位置是独立的 scoring(评分) 或 intelligence(智能分析) 层：

- 输入来自 JCVI 结果、注释信息、序列特征、启动子区域、同源关系等。
- 输出写入 run summary 和报告目录。
- 与基础共线性分析解耦，避免评分模型不可用时影响核心分析。
- 将来可被 GUI、插件和 Agent 共同使用。

当前 run summary 中已经预留了 scoring 结构，但正式机器学习模型还没有接入。这个预留是有意为之：先保证真实共线性主流程稳定，再逐步加入智能分析。

## 十二、CLI 思想重整

CLI 是项目的重要入口。我们在开发后期对 CLI 进行了几轮重整。

新的 CLI 思想是：

- 裸命令优先照顾零基础使用场景，尽量直接执行常见操作。
- 高级参数保留更多可控空间。
- `check` 裸命令默认输出整理后的人类可读格式。
- `check -j` 或 `check --json` 输出 JSON，风格接近 Linux 常用短参数。
- `analyze mcscan auto <input-dir> <output-dir>` 作为常用的一步式入口。
- `analyze run request.json` 作为高级入口、插件入口和未来 GUI/Agent 的统一入口。
- `help xxx` 与 `xxx --help` 走同一套帮助内容。

我们还为 CLI 增加了更友好的终端界面：

- 彩色标题和状态字段。
- 参数名与解释使用不同颜色。
- 整体采用低饱和、偏蓝色的马卡龙色调。
- 错误信息使用红色调。
- UI 渲染抽到独立层，避免污染功能代码。

在最近一轮清理中，我们进一步移除了旧的手动 CLI 方法：

- 移除 `--species`。
- 移除 `--raw-species`。
- 移除 `-i/--input`。
- 移除 `-o/--outdir`。
- 移除旧的手动参数分支和相关 helper。

现在普通 CLI 只保留清晰的一步入口：

```text
analyze mcscan auto <input-dir> <output-dir>
```

显式物种清单统一通过 request(JSON 请求)：

```text
analyze run request.json
```

这让命令体系更干净，也更适合未来接入 GUI、插件和 Agent。

## 十三、HAIant 插件适配

HAIant 插件最初参考平台示例进行接入。随着 GenomeLens CLI 架构变化，我们也同步调整插件。

早期插件会直接拼接旧 CLI 参数，例如 `--species`、`--raw-species`、`--outdir`。在新架构下，这种方式会让插件绕开统一 request 协议，形成第二套入口。因此我们将插件改为：

- 读取 HAIant 平台传入的 `params.json`。
- 解析相对路径，并按 `params.json` 所在目录处理。
- 生成 `genomelens_request.json`。
- 调用 `GenomeLens-runtime.exe analyze run genomelens_request.json`。

这样 HAIant 插件、CLI 高级入口和未来 GUI 都走同一个 request 协议，后续维护成本更低。

插件测试也做了改进：测试生成的 request 不再污染样例目录，而是写入 pytest 临时目录。这样测试更干净，也更符合持续集成环境。

## 十四、文档与注释工程

随着功能逐步增多，我们做了系统性的文档整理。文档语言统一采用自然中文表达，专业名词保留英文(中文)对照形式，例如 engine(引擎)、workflow(工作流)、manifest(JSON 清单)、summary(运行摘要)。

新增和调整过的文档包括：

- `docs/使用方法`：面向普通使用者，解释如何准备输入、运行分析、查看输出。
- `docs/开发手册`：面向开发维护，记录架构、构建、调试、工具链和迁移策略。
- `docs/项目介绍.md`：面向汇报和快速理解，介绍项目定位、能力边界和当前状态。
- `python/项目架构.md`：面向新人，梳理旧版 Python shell 的目录结构、执行流程、核心对象和阅读路线（该文档随布局迁移已归档，当前等价阅读路径为 `platform/src/genomelens/` 与 `platform/README.md`）。
- `engines/jcvi/上游修改汇总.md`：记录 vendored JCVI 相对上游的本地改动。
- `docs/开发手册/架构调整`：集中存放长期架构演进方案。

我们多次强调文档必须与真实实现一致。例如当前实现是基础多物种 pairwise 编排，而不是已经完成一张跨所有物种的最终美化总图；当前 scoring 是预留结构，而不是已经接入机器学习模型。这个口径对后续汇报和交付很关键。

## 十七、开发过程中的关键判断

我们在过程中做过几次重要判断。

第一，GenomeLens 不能只是“命令拼接器”。它需要形成稳定的 request、manifest、summary 协议，这样才能支持 CLI、GUI、插件和 Agent。

第二，JCVI 结果必须真实。简化算法可以用于诊断或未来快速预览，但不能在正式流程中冒充可发表结果。

第三，多物种流程应先基于双物种真实流程扩展。这样可以先获得稳定的基础能力，再逐步建设全局优化和美化图。

第四，Windows first 不等于 Windows only。当前交付优先 Windows，但核心设计必须避免把 `.exe`、路径分隔符、脚本形式和打包方式硬编码进业务逻辑。

第五，GUI 不应该复制分析逻辑。Tauri GUI 应该调用同一个核心协议，而不是另写一套工作流。

第六，文档必须写真实状态。项目还没完成的内容要明确写成未完成，已经完成的内容要说明测试和边界。

第七，旧接口要及时移除。只保留“兼容但没人该用”的旧入口，会让新人、插件和未来 GUI 不断走错路。

## 十八、一次典型开发节奏

我们的日常开发逐渐形成了这样的节奏：

1. 先围绕一个问题确认真实状态，例如是否接入 JCVI 全流程、是否支持多物种、是否还存在旧 CLI。
2. 读取相关源码、测试和文档，而不是只看单个文件。
3. 将问题拆成实现、测试、文档和交付风险几部分。
4. 修改代码时尽量保持边界清晰，不把 UI、配置、算法、工具链混在一起。
5. 每次功能变化同步更新文档口径。
6. 跑最相关的测试，再跑更大的测试集合。
7. 用 `git diff --check` 和搜索确认没有格式问题和旧内容残留。
8. 用 Git 提交阶段性成果。
9. 对还没完成的部分写清楚边界和后续计划。

这个节奏让项目在不断变化的目标中仍然保持可控。
