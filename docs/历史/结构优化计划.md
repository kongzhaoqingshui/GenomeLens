## 当前问题清单

通过本次整理，已发现以下结构性短板：

1. **`workflow_controller.py` 过大**
   - 原文件 657 行，包含 pairwise、reference-vs-targets、multi-species 三种执行策略。
   - 长方法内部混合了状态机推进、工具链检查、输入预处理、结果归档、summary 组装。

2. **`analyze mcscan` 参数扁平**
   - 30+ 个参数一次性注册到 parser，没有分组，help 输出难以阅读。
   - 已在本轮通过 `add_argument_group` 解决。

3. **跨层传递裸 `dict[str, object]`**
   - `run_summary`、`check payload`、`engine_result` 等仍用无 schema 的字典。
   - 字段名依赖字符串约定，容易出现拼写错误和 UI/引擎不同步。

4. **`request_normalizer.py` 职责过重**
   - 同时处理目录发现、参考物种解析、配置合并、CLI 默认值回退。
   - 单文件长度超过 400 行。

5. **配置与请求模型边界模糊**
   - `ConfigModel` 与 `AnalysisRequest` / `McscanMethodConfig` / `WorkflowOptions` 存在字段命名和类型差异。
   - 跨层转换时容易产生重复代码。

6. **文档重叠与过时**
   - 已在本轮按职责重新划分文档，但仍有部分内容需要随代码迭代继续清理。

---

## 优化目标

- 单一职责：每个模块/函数只做一件事。
- 强类型：逐步用 dataclass / TypedDict / Pydantic 替代裸字典。
- 可测试：减少长函数，增加纯函数辅助模块。
- CLI/引擎契约清晰：CLI 只负责参数解析，请求归一化与引擎适配由独立模块负责。

## 阶段化方案

### 短期（本轮已部分落地）

| 改动 | 路径 | 状态 |
|------|------|------|
| 将 `WorkflowController` 拆分为 runner 子模块 | `platform/src/genomelens/app/controller/runners/` | ✅ 已完成 |
| 提取共享 helper（artifact、summary、ui block） | `platform/src/genomelens/app/controller/runners/_shared.py` | ✅ 已完成 |
| `analyze mcscan` 参数分组 | `platform/src/genomelens/cli/commands/analyze.py` | ✅ 已完成 |

短期剩余：

- 为 `McscanSummary` / `CheckReport` 引入 TypedDict 或 dataclass。
- 将 `render_analysis_summary` / `render_check_report` 中依赖裸字典的字段访问改为 dataclass 属性访问。

### 中期

1. **拆分 `request_normalizer.py`**
   - `input_resolver.py`：输入目录发现、文件配对、GFF/CDS/BED 模式识别。
   - `reference_resolver.py`：参考物种名称/索引解析、物种重排。
   - `option_merger.py`：CLI 参数、config 默认值、环境变量的优先级合并。
   - `request_assembler.py`：把解析结果组装成 `AnalysisRequest`。

2. **引入强类型 summary**
   - `platform/src/genomelens/core/summary_models.py`
   - 定义 `McscanRunSummary`、`PairwiseJobSummary`、`CheckReport`。
   - UI renderers 接收这些类型，而非 `dict[str, object]`。

3. **统一跨层字段命名**
   - 让 `McscanMethodConfig`、`McscanRequest`、`WorkflowOptions` 中的同源字段名称一致。
   - 在 adapter 层做一次性映射，而不是多处散落的转换。

### 长期

1. **引入 Pydantic / JSON Schema**
   - 为 engine manifest、run_summary、config 定义 schema。
   - 实现 engine → shell 的强类型契约校验。

2. **CI / 类型检查**
   - 添加 mypy 或 pyright 配置。
   - GitHub Actions 运行 ruff + pytest + 类型检查。

3. **错误码与国际化**
   - 统一错误码枚举。
   - 用户文案集中管理，便于后续国际化。

---

## 验收标准

- 每个子模块行数不超过 400 行；函数不超过 60 行。
- 新增/修改后测试覆盖率不下降。
- `ruff check` 全绿；`pytest platform/tests engines/jcvi/tests` 全绿。
- 裸 `dict[str, object]` 的使用点逐轮减少，并在 `项目改进意见.md` 中跟踪。
