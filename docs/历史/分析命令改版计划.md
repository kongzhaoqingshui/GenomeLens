# 分析命令改版计划（已归档）

> **归档说明**：本文档已完全实施。当前仓库中 `analyze run <request.json>`、`analyze template mcscan`、`analyze schema` 与 `analyze mcscan jcvi` 子命令均已落地，旧 `python/` 路径已迁移为 `platform/`。

---

> **历史文档提示**：本计划已落地。当前仓库中，旧 `python/` 路径已迁移为 `platform/`，`analyze mcscan auto` / `analyze run request.json` / `analyze template mcscan` 均已实现。文中出现的 `python/` 路径请对应到 `platform/`。

## 文档目的

这份文档规划 `genomelens analyze` 系列命令的下一轮改版。目标是让命令对新手更直观、对高级用户更可控、对 GUI/Agent/插件更稳定，同时为后续平台化大重构留出干净边界。

本计划只描述分析命令层，不替代 engine(引擎) 协议、JCVI workflow(工作流) 接入规则和多平台兼容方案。

## 当前问题

当前 `analyze mcscan` 已能运行真实 JCVI 共线性流程，并支持：

- `auto` 目录自动发现。
- 显式物种清单可通过 `analyze run request.json` 输入。
- 主配置 `genomelens.config.json` 与 JCVI 子配置 `jcvi.config.json`。
- 2 到 n 个物种 all-vs-all pairwise(全组合两两比较) 编排。

但命令形态仍有几个结构性问题：

- `analyze mcscan` 同时承担“选择方法”和“执行方法配置”，后续方法变多时会膨胀。
- `auto` 与 JSON request(JSON 请求) 需要在内部保持同一条执行路径。
- 未来 Tauri GUI、HAIant plugin(智然体插件)、AI Agent 都需要稳定 request(JSON 请求) 入口，不能长期依赖拼接 CLI 参数。
- 用户心智应从“记一堆参数”转向“启动分析、选择方法、确认配置、执行任务”。

## 改版目标

### 用户目标

- 裸命令适合无基础用户，能通过提示完成任务。
- 简洁命令适合常见自动流程，一行完成分析。
- 参数命令适合熟悉用户，保留可控空间。
- JSON request(JSON 请求) 适合 GUI、插件、Agent 和批处理。

### 架构目标

- 所有分析入口最终归一化为同一个 `AnalysisRequest`。
- 所有分析方法通过 `AnalysisDispatcher(分析调度器)` 分发。
- 方法配置使用统一命名：`method`、`input`、`output`、`config`、`options`。
- CLI 不直接承载某个 engine(引擎) 的私有协议，而是生成平台核心 request。
- 当前 `McscanRequest` 作为 `AnalysisRequest.method_config` 的一种具体载荷逐步下沉。

## 目标命令形态

### 1. 裸命令：交互式分析入口

```powershell
genomelens analyze
```

行为：

1. 显示可用 analysis method(分析方法)。
2. 让用户选择 `mcscan` 等方法。
3. 询问输入来源：目录自动发现、手动文件、JSON request。
4. 询问输出目录。
5. 显示即将执行的配置摘要。
6. 用户确认后运行。

适合对象：无基础用户、第一次运行、终端工作台。

### 2. 简洁命令：自动流程

```powershell
genomelens analyze mcscan auto input output
```

行为：

- 扫描 `input` 目录。
- 自动识别同名 `.bed/.cds` 或 `.gff/.fa` 文件对。
- 自动生成 `AnalysisRequest`。
- 使用默认参数或配置文件参数运行。

适合对象：大多数命令行用户。

### 3. JSON 命令：显式控制

```powershell
genomelens analyze run request.json
```

行为：

- 读取完整 `AnalysisRequest`。
- 校验 schema version(JSON 结构版本)。
- 分发到对应 method runner(方法运行器)。

适合对象：熟悉参数的用户、脚本用户。

### 4. 模板命令：生成 request

```powershell
genomelens analyze template mcscan > mcscan.request.json
```

行为：

- 输出可编辑的 request(JSON 请求) 模板。
- 用户可在模板中显式填写 species(物种)、配置和运行选项。

适合对象：Tauri GUI、HAIant plugin、AI Agent、批处理系统。

### 5. 已移除的旧 CLI 方法

`--species` 与 `--raw-species` 不再作为 CLI 参数暴露。显式物种文件清单统一写入 request(JSON 请求)，再通过 `analyze run` 执行。

行为：

- CLI 保持短而直观。
- 平台、GUI、插件和 Agent 共用同一种 request 协议。

适合对象：后续维护者理解命令收口边界。

## 统一 request 结构

目标结构如下：

```json
{
  "schema_version": 1,
  "kind": "analysis_request",
  "method": "mcscan",
  "input": {
    "mode": "auto_directory",
    "directory": "input",
    "species": []
  },
  "output": {
    "directory": "output",
    "force": false,
    "formats": ["png"]
  },
  "config": {
    "project_config": "workspace/genomelens.config.json",
    "method_config": "workspace/jcvi.config.json"
  },
  "options": {
    "preset": "auto",
    "threads": null,
    "min_block_size": null,
    "workflow": "graphics_synteny"
  }
}
```

### 字段说明

- `schema_version`：request schema(JSON 请求结构) 版本。
- `kind`：固定为 `analysis_request`，避免与 config、summary 混淆。
- `method`：分析方法，例如 `mcscan`。
- `input`：输入来源与物种文件描述。
- `output`：输出目录、覆盖策略、格式。
- `config`：工具主配置与方法配置路径。
- `options`：该方法的通用运行选项。

### 输入模式

`input.mode` 先支持三类：

- `auto_directory`：目录自动发现。
- `prepared_species`：显式 BED+CDS。
- `raw_species`：显式 GFF+FASTA。

未来可扩展：

- `manifest`：读取物种 manifest(清单)。
- `dataset_id`：GUI 或平台数据集 ID。
- `remote_uri`：远端对象存储路径。

## 内部模块规划

建议新增或重组为以下模块：

```text
python/src/genomelens/analysis/
  request_models.py
  request_loader.py
  request_normalizer.py
  dispatcher.py
  methods/
    mcscan.py
    registry.py
```

### request_models.py

职责：

- 定义 `AnalysisRequest`、`AnalysisInput`、`AnalysisOutput`、`AnalysisConfigRef`、`AnalysisOptions`。
- 不导入 CLI。
- 不导入具体 engine。

### request_loader.py

职责：

- 从 JSON 文件读取 request。
- 做 schema version 检查。
- 把路径规范化，但不启动分析。

### request_normalizer.py

职责：

- 把 CLI 参数、交互式回答、JSON request 统一转为 `AnalysisRequest`。
- 把 `auto_directory` 解析成 species 列表。
- 把配置文件默认值补齐到 request。

### dispatcher.py

职责：

- 根据 `request.method` 找到 method runner。
- 统一处理状态、错误、日志和 summary(摘要)。
- 不知道某个方法内部如何调用 JCVI 或其他引擎。

### methods/mcscan.py

职责：

- 把 `AnalysisRequest(method=mcscan)` 转为当前 `McscanRequest`。
- 调用现有 `run_mcscan()`。
- 返回统一 `AnalysisResult` 或现有 run summary。

### methods/registry.py

职责：

- 声明可用方法。
- 提供方法元信息：名称、描述、输入模式、配置模板、是否稳定。

## CLI 层规划

CLI 只做四件事：

1. 展示帮助和交互提示。
2. 解析命令行参数。
3. 调用 request normalizer(请求归一化器)。
4. 调用 dispatcher(调度器)。

CLI 不应直接：

- 调用 `run_mcscan()`。
- 读取 engine manifest(引擎清单)。
- 拼接 engine 私有参数。
- 持有复杂业务分支。

## 迁移阶段

### 阶段 1：新增 request 模型，不改用户命令

目标：

- 保留当前 `analyze mcscan auto`。
- 新增 `AnalysisRequest` dataclass(数据类)。
- 当前 CLI 先归一化为 `AnalysisRequest`，再转回 `McscanRequest`。

验收：

- 现有 21 条 python 测试全过。
- 新增 request normalization(请求归一化) 单元测试。

### 阶段 2：新增 `analyze run request.json`

目标：

- 建立机器入口。
- GUI/插件/Agent 后续优先接入该入口。

验收：

- `genomelens analyze run request.json` 能跑通 BED+CDS 双物种样例。
- `request.json` 中缺字段时给出清晰错误。

### 阶段 3：新增 `analyze template mcscan`

目标：

- 降低 JSON request 的使用门槛。
- 固化字段命名。

验收：

- 模板可被 `analyze run` 读取。
- 文档示例与模板字段一致。

### 阶段 4：交互式 `genomelens analyze`

目标：

- 裸命令提供向导式体验。
- 不污染核心分析代码。

验收：

- 可选择 method(方法)。
- 可选择 auto directory(目录自动发现)。
- 可确认配置摘要并执行。
- 可取消，不产生半成品输出。

### 阶段 5：收口旧 CLI 内部路径

目标：

- `analyze mcscan auto` 与 `analyze run request.json` 全部走同一个 dispatcher。
- CLI 层不再直接构造 `McscanRequest`。

验收：

- 搜索 `run_mcscan(`，CLI 层不再直接调用。
- 所有分析命令都有 request 快照写入输出目录。

## 命名规则

### 命令命名

- `analyze`：分析入口。
- `method`：分析方法。
- `run`：运行 JSON request。
- `template`：生成 request 模板。
- `auto`：目录自动发现。

### JSON 字段命名

统一使用 snake_case：

- `schema_version`
- `method`
- `input`
- `output`
- `config`
- `options`
- `species`
- `input_mode`
- `project_config`
- `method_config`

### 避免使用

- 避免长期把顶层字段命名为 `query` / `subject`。
- 避免把 `jcvi_*` 字段放在平台通用 request 顶层。
- 避免让 GUI 或插件生成 CLI 字符串作为主集成方式。

## 与当前 `mcscan` 的关系

当前 `McscanRequest` 不需要马上删除。它在过渡期作为 `mcscan method runner(mcscan 方法运行器)` 的内部请求对象。

目标关系：

```text
CLI / GUI / Plugin / Agent
  -> AnalysisRequest
  -> AnalysisDispatcher
  -> McscanMethodRunner
  -> McscanRequest
  -> run_mcscan()
```

这样既不打断当前真实 JCVI 链路，也能给未来其它方法留位置。

## 与配置系统的关系

配置继续保持两层：

- `genomelens.config.json`：工具本身配置。
- `jcvi.config.json`：JCVI 方法配置。

在 request 中只引用路径，不把配置文件全文复制进 request。运行时可以把最终解析后的配置快照写入输出目录：

```text
<outdir>/inputs/analysis_request.json
<outdir>/inputs/resolved_config.json
```

这样便于审计，也便于发表图件追溯。

## 测试计划

### 单元测试

- `AnalysisRequest` JSON roundtrip(往返)。
- CLI 参数到 request 的归一化。
- `auto_directory` 文件发现。
- 缺少输入目录、少于两个物种、混合输入模式的错误。
- config 默认值合并。

### 集成测试

- `analyze mcscan auto input output`。
- `analyze run request.json` 中显式 species(物种) 清单。
- `analyze run request.json`。
- `analyze template mcscan | analyze run`。
- 三物种 all-vs-all pairwise。

### 文档测试

- 文档示例命令至少有 smoke test(冒烟测试) 覆盖。
- request 模板与文档字段一致。

## 风险与约束

- 不能为了新命令形态重写 JCVI 核心流程。
- 不在 CLI 层实现复杂业务状态机。
- 不提前承诺“全局多物种最终美化图”已完成。
- 不恢复旧配置兼容路径。
- 不把 HAIant plugin 参数模型反向塞进平台核心。

## 不做项

本轮命令改版不直接实现：

- 新的 JCVI 算法能力。
- 全局多物种 layout(布局) 优化。
- 机器学习评分。
- Tauri GUI。
- 云端任务调度。

这些能力应在统一 `AnalysisRequest` 和 `AnalysisDispatcher` 稳定后逐步接入。

## 推荐优先级

P0：

- 新增 `AnalysisRequest` 模型。
- 让当前 `analyze mcscan auto` 先归一化为 request。
- 输出 `<outdir>/inputs/analysis_request.json`。

P1：

- 新增 `analyze run request.json`。
- 新增 `analyze template mcscan`。
- CLI 层不再直接调用 `run_mcscan()`。

P2：

- 实现裸 `genomelens analyze` 交互向导。
- 加入配置摘要确认。
- 为 Tauri GUI 和 Agent 固化 request schema。

P3：

- 将 HAIant plugin 改为生成 request JSON，而不是拼接 CLI 参数。
- 为未来其它比较基因组学方法接入 registry(注册表)。
