## 仍存在的短板

1. **跨层 summary 仍未完全强类型化**
   - `run_summary`、`check payload`、`engine_result`、runner 内部的 `pairwise_jobs` 等仍大量依赖 `dict[str, object]`。
   - 字段名依赖字符串约定，UI render 与引擎返回容易出现不同步。
   - `platform/src/genomelens/app/controller/runners/` 内仍存在多处 `list[dict[str, object]]` 临时结构。

2. **`request_normalizer.py` 职责过重**
   - 单文件同时处理目录发现、参考物种解析、配置合并、CLI 默认值回退、请求组装。
   - 尚未拆分为 `input_resolver`、`reference_resolver`、`option_merger`、`request_assembler`。

3. **配置与请求模型边界仍有些模糊**
   - `ConfigModel`、`McscanMethodConfig`、`McscanRequest`、`WorkflowOptions` 中存在命名和类型差异。
   - 部分字段在 adapter 层多处转换，没有统一映射点。

4. **HAIant 插件临时拷贝目录方案有代价**
   - 当前通过拷贝物种文件到临时目录来兼容 `analyze mcscan` 自动发现，对于大型基因组文件会增加磁盘 I/O。
   - 未来应恢复稳定的 JSON request 入口，或让 `analyze mcscan` 支持显式物种清单，避免拷贝。

5. **缺少自动化 CI / 类型检查**
   - 尚未配置 mypy / pyright。
   - 没有 GitHub Actions 工作流自动运行 ruff + pytest。
   - 构建产物（`app/`）未在 CI 中验证。

6. **错误码与文案未统一**
   - 错误信息分散在 CLI、adapter、runners、plugin 中，未集中管理。
   - 没有错误码枚举，不利于 GUI/插件做程序化错误处理。

7. **引擎能力文档与实现需持续同步**
   - 新增 `local_synteny`、`graphics_karyotype_global` 后，文档已更新，但随着能力继续扩展，需保持 `engines/jcvi/README.md`、`docs/CLI.md`、`docs/使用方法/README.md` 同步。

## 后续改进优先级

### P0：summary 强类型化

- 在 `platform/src/genomelens/core/summary_models.py` 定义 `McscanRunSummary`、`PairwiseJobSummary`、`CheckReport` dataclass 或 TypedDict。
- 先让 `render_analysis_summary`、`render_check_report` 消费强类型对象；再逐步替换 runner 内部字典组装。
- 目标：3 个月内让 `dict[str, object]` 在业务代码中减少 50% 以上。

### P1：CI 与类型检查

- 添加 `mypy` 或 `pyright` 配置，先对 `platform/src/genomelens/core/`、`runners/`、`data/config/` 开启严格类型检查。
- 新增 GitHub Actions 工作流：
  - `ruff check` + `ruff format --check`
  - `pytest platform/tests engines/jcvi/tests integrations/haiant_plugin/tests`
  - 可选：构建 `app/` 并执行 `--help` / `analyze mcscan --help` 烟测。

### P2：拆分 `request_normalizer.py`

- `input_resolver.py`：目录发现、文件配对、GFF/CDS/BED 模式识别。
- `reference_resolver.py`：参考物种名称/索引解析、物种重排。
- `option_merger.py`：CLI 参数、config 默认值、环境变量优先级合并。
- `request_assembler.py`：组装成 `AnalysisRequest`。

### P3：统一跨层字段命名

- 让 `McscanMethodConfig`、`McscanRequest`、`WorkflowOptions` 中的同源字段名称一致。
- 在 `JcviEngineAdapter` 内做一次性字段映射，而不是多处散落转换。

### P4：恢复稳定的 JSON request 入口（长期）

- 当需要 GUI/插件/Agent 稳定调用时，重新设计 `analyze run` 或 `analyze request` 入口。
- 入口必须基于已定义的 JSON Schema / Pydantic 校验，避免再次出现命令与内部模型脱节。

### P5：错误码与国际化

- 定义错误码枚举（`ERROR_REQUEST_INVALID`、`ERROR_TOOLCHAIN_MISSING`、`ERROR_ENGINE_FAILED` 等）。
- 用户文案集中到一个 `messages.py` 模块，便于后续国际化。
