# GenomeLens 项目开发规范

> 本规范是项目级总纲，覆盖代码、测试、协作、发布与文档。具体细则请对照：
>
> - 第 4 节“代码风格”：Python 格式、region、docstring、import 等细节。
> - `协作开发方案.md`：Git 分支模型、PR 流程、`main` 保护规则。
> - `能力接入规则.md`：新增 engine workflow、多物种能力扩展的联动要求。

---

## 1. 适用范围与目标

本规范适用于项目自有的全部源码与测试：

- `platform/src/genomelens/` 与 `platform/tests/`
- `engines/jcvi/src/jcvi_genomelens/` 与 `engines/jcvi/tests/`
- `integrations/haiant_plugin/`（按该模块约定执行）
- `docs/`、`scripts/`、`.claude/skills/`

**不适用** vendored 的上游代码，例如 `references/upstream/jcvi/`、`engines/jcvi/src/jcvi/`。

目标：在 2~3 人核心团队 + 外部贡献者的规模下，保证 `main` 分支随时可发布、代码可审查、功能可回溯。

## 2. 开发环境

### 2.1 Python 环境

统一使用 `genomelens` conda 环境，解释器版本 **Python 3.12**。

```powershell
conda activate genomelens
python --version
```

项目级默认环境已配置：

- Claude Code：`.claude/settings.local.json` 已把 `genomelens` 环境目录前置到 `PATH`，并设置 `CONDA_DEFAULT_ENV`。
- VS Code：`.vscode/settings.json` 已指定 `python.defaultInterpreterPath`。

### 2.2 可编辑安装

`platform/` 与 `engines/jcvi/` 必须分别做 editable install：

```powershell
pip install -e ./platform
pip install -e ./engines/jcvi
```

注意：旧 editable 安装可能指向 `D:\GenomeLens\python\src` 等历史路径，若出现导入异常，应重新安装并检查 `pip show genomelens`、`pip show jcvi-genomelens` 的 Location。

### 2.3 外部工具链

- BLAST+：`blastn`、`makeblastdb` 需在 PATH 中，或在 `jcvi.config.json` 中显式配置路径。
- 可选 LAST：`lastal`、`lastdb` 仅在 `align_soft == "last"` 时需要。
- 绘图依赖由 `engines/jcvi` 的 `pyproject.toml` 管理。

## 3. 仓库结构与模块职责

```text
GenomeLens/
├── platform/                 # shell（外壳）：CLI、配置、调度、错误处理、结果汇总
├── engines/jcvi/             # 第一个 engine（引擎）：持有 vendored JCVI，执行具体 workflow
├── engines/                  # 未来可平级接入第二个比较基因组学引擎
├── integrations/haiant_plugin/  # 智然体插件入口，负责把插件参数翻译为 shell 请求
├── gui/、agents/             # Tauri GUI 与 Agent 预留目录（当前占位）
├── references/               # 示例数据与本地参考
├── toolchains/               # 本地运行时缓存，不进入 Git
├── docs/                     # 用户文档、开发手册、更新计划
└── scripts/                  # 构建与辅助脚本
```

核心边界：

- `platform` **不得直接导入 `jcvi`**；所有对 engine 的调用通过外部进程 + manifest 完成。
- engine 内部可以调用 vendored JCVI，但新增 workflow 必须同步暴露到 `probe` 与 dispatcher。

## 4. 代码风格

本规范适用于 project-owned 代码，即：

- `platform/src/genomelens/`
- `engines/jcvi/src/jcvi_genomelens/`
- `integrations/haiant_plugin/src/`

不包含 vendored 的 JCVI 代码（`references/upstream/jcvi/`、`engines/jcvi/src/jcvi/` 等）。

自动化基线为 [Ruff](https://docs.astral.sh/ruff/)，配置见各模块的 `pyproject.toml`：

- `line-length = 120`
- `target-version = "py312"`
- lint 规则：`E`、`F`、`I`、`UP`、`B`

Ruff 无法覆盖的风格细节（region 注释、docstring 标点、空行语义）由本节约束。

### 4.1 文件头与模块 docstring

每个 Python 文件第一行必须是模块级 docstring，简述该文件职责：

```python
"""CLI 展示层：集中管理颜色、字段和交互提示"""
```

- 单行即可，**末尾不加句号**。
- 需要时可在 docstring 里用括号标注中英术语，例如 `workflow(工作流)`。

### 4.2 Imports

#### 必须包含 future annotations

```python
from __future__ import annotations
```

#### 分组与顺序

1. 标准库
2. 第三方库
3. 项目内部模块

组与组之间空一行，组内按字母顺序排序（Ruff `I` 规则自动处理）。

#### 用 region 包裹 import 段

```python
# region import
from __future__ import annotations

import argparse
import os

from genomelens.analysis.workflows.registry import list_workflow_plugins

# endregion
```

- `# region import` 后不加空行。
- 距离行尾空一行接 `# endregion`；`# endregion` 后空两行再接后续代码。

### 4.3 Region 注释

按功能把相关函数/类归到同一个 region 块中。

```python
# region 内部格式操作函数
def _supports_color(...) -> bool:
    ...


def _paint(...) -> str:
    ...

# endregion


# region 工作台与提示相关函数
def render_workbench_banner(...) -> str:
    ...

# endregion
```

格式要求：

- `# region 中文描述`：描述末尾**不加标点**。
- region 内函数之间空一行。
- `# endregion` 独占一行，前面空一行。
- `# endregion` 之后空两行再接下一个顶级定义（函数/类/region）。
- 避免 region 嵌套过深，一般只用于 `import`、同类工具函数、命令注册分组、帮助函数等。

### 4.4 Docstring

#### 单行 docstring

用于简单函数/类，**末尾不加句号**：

```python
def _paint(text: str, color: str, *, enabled: bool) -> str:
    """按需为文本包裹 ANSI 颜色"""
```

#### 多行 docstring

复杂函数使用多行，结构为：

```python
def run(manifest: EngineRunManifest, outdir: str | Path) -> tuple[list[CommandAudit], dict[str, object]]:
    """运行目标基因局部共线性分析

    步骤：
    1. 运行 pairwise MCscan 得到全基因组 blocks。
    2. 按目标基因上下游窗口截取局部 blocks。
    3. 生成局部 layout。
    4. 调用 JCVI graphics.synteny 出图。
    """
```

- 第一行是简短摘要，**不加句号**。
- 空一行后写详细说明。
- 详细说明内部可使用列表、段落。

#### 中英术语

允许在 docstring 中用括号标注英文/中文术语，帮助跨背景读者：

```python
"""生成整理后的 analyze(分析) 文本摘要"""
```

### 4.5 行注释

行注释解释“为什么”或“注意事项”，而非重复代码字面含义。

```python
# 边框内宽度 = max(显式最小宽度, 所有行的可见宽度)
inner = max([min_width, *(_visible_width(line) for line in lines)]) if lines else min_width

# 用户可以通过环境变量强制关闭/开启颜色
if os.environ.get("NO_COLOR"):
    return False
```

- 行尾注释与代码之间留 **两个空格**。
- 行注释末尾**不加句号**。
- 对复杂正则或状态机分支，推荐在逻辑段前加块注释说明。

#### 类字段行尾注释

数据类（dataclass）、配置模型、请求模型等公共类的字段，推荐在同一行用中文行尾注释说明语义，便于阅读与后续生成文档。

- 注释以中文为主，必要时保留英文/中文术语对照。
- 同一注释组内可对齐，使 `#` 起点大致一致；当出现以下情况时另起一段对齐：
  - 字段类型或默认值长度差异明显（例如从 `str` 跳到 `list[PortDeclaration] = field(...)`）。
  - 字段语义分组发生变化（例如从“引擎元数据”切换到“产物字段”）。
  - 组之间用空行或分组注释隔开。
- 如果手动对齐后的结果与 Ruff format 默认行为冲突，用 `# fmt: off` / `# fmt: on` 保护该字段区域；标记应放在类体缩进位置，并包住整组字段。

示例：

```python
@dataclass(frozen=True)
class SubModuleSpec:
    """子模块规范：单个可编排子模块的完整元数据"""

    # fmt: off
    module_id: str                      # 子模块唯一标识（如 jcvi.graphics_dotplot）
    name: str                           # 面向用户展示的子模块名称
    description: str                    # 子模块功能描述
    category: SubModuleCategory         # 职责分类（计算/渲染/混合/其他）
    domain: SubModuleDomain             # 领域分类（共线性/通用）
    module_kind: SubModuleKind          # 子模块编排类型（lightweight/aggregate）
    engine_workflow: str                # 映射到底层引擎的 workflow 名称
    standalone: bool                    # 是否可独立运行

    # 端口、参数与标签
    inputs: list[PortDeclaration] = field(default_factory=list)           # 输入端口声明
    outputs: list[PortDeclaration] = field(default_factory=list)          # 输出端口声明
    parameters: list[ParameterDeclaration] = field(default_factory=list)  # 可调参数声明
    labels: list[str] = field(default_factory=list)                       # 可挂载的标签列表
    # fmt: on
```

> 不必追求绝对严格的空格数，保持“自然、可读、同组对齐”即可。

### 4.6 空行

#### 顶级定义之间

函数与函数、类与类、region 与 region 之间空 **两行**。

#### region 内函数之间

空 **一行**。

#### 函数内逻辑段之间

根据语义空行，把函数拆成“输入准备 → 核心计算 → 返回组装”等段落。

### 4.7 命名与类型

#### 命名

- 函数/变量：`snake_case`
- 类：`PascalCase`
- 常量：模块级全大写 `SCREAMING_SNAKE_CASE`
- 私有辅助函数/变量：以下划线开头 `_helper`

#### 类型提示

- 必须为主函数添加类型提示。
- 使用 `|` 联合类型：`str | None`、`list[str] | None`。
- 跨层 JSON 结构当前允许 `dict[str, object]`，但应逐步替换为 `TypedDict` 或 dataclass。

#### 函数长度

- 目标：函数不超过 60 行。
- 超过 100 行的函数必须拆分或提取辅助函数。

### 4.8 字符串与格式化

- 优先使用 f-string。
- 字符串引号：代码库统一使用双引号 `""`（Ruff format 会处理）。
- 多行字符串使用括号隐式连接或三引号 docstring，避免反斜杠续行。

### 4.9 错误处理

- 捕获通用异常时，必须添加 `# noqa: BLE001` 与说明注释。
- CLI/引擎边界允许兜底捕获，但应转换为 `format_user_error` 等用户可读错误。

### 4.10 自动化检查

本地提交前建议运行：

```powershell
ruff check platform/src platform/tests engines/jcvi/src/jcvi_genomelens engines/jcvi/tests integrations/haiant_plugin/src integrations/haiant_plugin/tests
ruff format --check platform/src platform/tests engines/jcvi/src/jcvi_genomelens engines/jcvi/tests integrations/haiant_plugin/src integrations/haiant_plugin/tests
pyright platform/src/genomelens engines/jcvi/src/jcvi_genomelens integrations/haiant_plugin/src
python -m pytest platform/tests engines/jcvi/tests integrations/haiant_plugin/tests
```

根目录已配置 `.pre-commit-config.yaml` 时，可通过 `pre-commit run --all-files` 自动执行。

## 5. 类型系统与静态检查

### 5.1 必须添加类型提示

- 公开函数、类方法、dataclass 字段必须标注类型。
- 优先使用 `|` 联合类型：`str | None`、`list[str]`。
- 跨层 JSON 中间结构当前允许 `dict[str, object]`，但应逐步迁移到 `TypedDict` 或 dataclass。

### 5.2 Pyright

- 平台模块启用 `pyright`，配置位于 `platform/pyproject.toml`：`typeCheckingMode = "basic"`。
- engine 模块目前未强制启用，但新增代码应保持类型正确，为后续开启做准备。
- 提交前建议运行：

```powershell
pyright platform/src/genomelens
```

### 5.3 常见类型窄化模式

`dict.get(...)` 返回值类型为 `object`，禁止直接假设为具体类型。推荐写法：

```python
raw_index = event.payload.get("index")
if raw_index is None:
    raise ValueError("payload missing 'index'")
index = cast(int, raw_index)
```

- 对可选的 dataclass 字段（如 `manifest.query: GenomeSpec | None`），先进行 `is None` 守卫再访问成员。
- 禁止用 `assert` 做运行时校验；使用显式 `raise` 并附带上下文信息。

## 6. 测试

### 6.1 测试结构

```text
platform/tests/unit/           # 单元测试，不依赖外部 engine
platform/tests/integration/    # 集成测试，可能调用 jcvi-genomelens engine
engines/jcvi/tests/unit/       # engine 单元测试
engines/jcvi/tests/integration/# engine 集成测试
integrations/haiant_plugin/tests/  # 插件测试
```

### 6.2 运行命令

```powershell
python -m pytest platform/tests
python -m pytest engines/jcvi/tests
python -m pytest integrations/haiant_plugin/tests
```

### 6.3 测试原则

- 新增功能必须伴随测试；修复 bug 应先写能复现失败的测试。
- 单元测试隔离外部依赖；需要真实 engine 的测试加 `pytest.mark.real_engine`。
- 集成测试的烟测场景应覆盖：
  - BED+CDS 双物种输入
  - GFF+FASTA 双物种输入
  - `graphics_synteny`、`graphics_dotplot`、`graphics_karyotype`、`pairwise`（含 `emit_ortholog`）、`local_synteny`
  - 多物种 all-vs-all pairwise 与全局核型总图
  - 智然体插件参数翻译

### 6.4 测试数据

- 示例数据放在 `references/` 下，避免把大文件放入 Git。
- 测试用临时目录优先使用 `tmp_path` fixture，并在测试结束时自清理。

## 7. 架构与接口规范

### 7.1 Shell-Engine 契约

- `platform` 通过 `JcviEngineAdapter` 写出 `task_manifest_v2` JSON，调用 `jcvi-genomelens run --manifest <path> --outdir <dir>`。
- manifest 必须包含 `schema_version`、`workflow`、`task`、`species`、`query`、`subject`、`toolchain`、`options`、`expected_outputs`、`meta`。
- engine 运行后必须写出 `engine_run_summary.json`，字段见 `platform/src/genomelens/engines/jcvi/models.py` 的 `JcviRunResult`。

### 7.2 Engine 探测

- engine `probe --json` 返回的 `capabilities` 只应包含当前真实可调度的 workflow。
- 随包但尚未暴露为 workflow 的 JCVI 模块应写入 `bundled_jcvi_modules`，不得混入 `capabilities`。

### 7.3 新增 Workflow 的联动清单

参考 `能力接入规则.md`，新增 workflow 时必须同步：

- engine dispatcher 与 probe
- shell workflow 校验与 adapter 映射
- HAIant 插件参数选项
- 最小样例与集成测试
- 用户文档与开发文档

### 7.4 多物种能力

- shell 层继续以 `species[]` 作为公开输入模型。
- engine 层 pairwise worker 以 manifest v3 的 `inputs.species[0:2]` 接收输入；`query/subject` 只可作为 engine 内部局部对象名，不应重新暴露为用户入口或 manifest 顶层字段。
- 中间结果按物种对归档并汇总，summary 明确记录每个物种、每个物种对、每个图件与最终总图状态。

## 8. 错误处理与日志

### 8.1 错误分层

- 内部异常使用项目自定义异常，例如 `EngineRunError`、`EngineProbeError`、`SummaryParseError`。
- CLI 边界应把异常转换为面向用户的可读错误，避免直接抛出 traceback。

### 8.2 日志

- 默认 `log_level = INFO`，`verbose` 为 false。
- 控制台默认静默，只输出启动句段与 WARNING/ERROR/CRITICAL（`ConciseConsoleFilter`）。
- 完整运行日志写入 `run.log`，结构化为 `task_started` / `task_finished` / `task_failed`。
- engine 步骤失败时记录 `argv`、`cwd`、`stderr` 摘要。

### 8.3 不推荐的做法

- 不要用裸 `except:` 或 `except Exception:` 吞掉异常；必须转换或重新抛出。
- 不要把敏感路径、密钥写入日志。

## 9. 配置与 CLI

### 9.1 配置来源优先级

命令行参数 > 请求 JSON > `jcvi.config.json` > 代码默认值。

### 9.2 默认值管理

- `jcvi.config.json` 初始化时应暴露所有用户可调项，包括 plot 优化开关、日志级别等。
- 新增 CLI 参数时，必须在 `analyze template` / JSON schema 中同步体现。

### 9.3 CLI 设计

- 子模块 CLI（`analyze submodule <module_id>`）与一站式工作流 CLI（`analyze workflow <workflow_id>`）由注册表元数据决定底层 workflow，避免与 `--jcvi-workflow` 冲突。
- 帮助文本保持中英术语对照，便于跨背景开发者阅读。

## 10. Git 工作流

详见 `协作开发方案.md`。关键要求：

- 只保留 `main` 一条长期分支，随时可发布。
- 新工作从 `main` 切出短生命周期分支，通过 PR 合并回 `main`，禁止直接 push。
- 分支命名：`feature/`、`fix/`、`hotfix/`、`docs/`、`refactor/`，全部小写、单词用 `-` 连接。
- 提交信息采用 [Conventional Commits](https://www.conventionalcommits.org/)：`type(scope): 简短描述`。
- PR 优先使用 **Squash and merge**，合并后删除远程功能分支。

## 11. 版本发布

### 11.1 语义化版本

遵循 [Semantic Versioning](https://semver.org/lang/zh-CN/)：

- `MAJOR`：不兼容的 CLI / 插件契约变更
- `MINOR`：向下兼容的功能新增
- `PATCH`：向下兼容的问题修复

当前处于 `v1.0.0-preview-1` 预览阶段，正式版 `v1.0.0` 将在核心功能全部补齐后发布。

### 11.2 发布 checklist

1. 确认 `main` 上所有检查通过（`ruff`、`pyright`、`pytest`）。
2. 更新 `platform/src/genomelens/_version.py` 与 `engines/jcvi/src/jcvi_genomelens/_version.py`（如适用）。
3. 在 `docs/更新计划/更新日志.md` 新增 `[X.Y.Z]` 章节。
4. 把 `docs/更新计划/计划更新的内容.md` 中已完成的条目归档到 `docs/历史/计划更新的内容.md`。
5. 当版本包含 GUI 预览时，确认 `gui/tauri` 的 `pnpm lint`、`pnpm typecheck`、`pnpm build:web` 与 `cargo check --manifest-path src-tauri/Cargo.toml` 通过。
6. 当版本提供 Docker 开发环境时，确认 `docker build -t genomelens-dev:<version> .` 构建成功，并执行 `docker run --rm genomelens-dev:<version> python -m pytest platform/tests -q` 烟测通过。
7. 创建并推送带注释标签：

```bash
git checkout main
git pull origin main
git tag -a vX.Y.Z -m "release: vX.Y.Z"
git push origin vX.Y.Z
```

6. 在 GitHub Releases 上填写发布说明。

- 规划文档必须“四节一致”：编号说明、版本分布、建议开发顺序、多人协作分工互相引用，不脱节。

---

## 13. 安全与依赖

- 不要把密钥、token、个人路径提交到仓库。
- `.claude/settings.local.json`、`.codex/`、`toolchains/`、`app/` 等本地文件不进入 Git。
- 新增依赖时，在对应模块的 `pyproject.toml` 中声明版本约束，并评估跨平台兼容性。
- vendored JCVI 的修改应单独记录补丁集（patchset），便于后续同步上游。
