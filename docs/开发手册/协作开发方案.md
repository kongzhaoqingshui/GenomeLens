# 协作开发方案

> 适用场景：2~3 名核心开发者 + 开源外部贡献者，以保护 `main` 分支为首要目标。

## 1. 分支模型：GitHub Flow（简化版）

```text
main  ──────────────────────────────────────►
       \
        feature/xxx  ──PR──►  main
       /
hotfix/xxx  ──PR──►  main
```

- 只保留一条长期分支 **`main`**，随时可发布。
- 所有新工作从 `main` 切出短生命周期分支。
- 通过 Pull Request（PR）合并回 `main`，禁止直接 `git push` 到 `main`。

## 2. 分支命名规范

| 类型 | 前缀 | 示例 |
|------|------|------|
| 新功能 | `feature/` | `feature/local-synteny-labels` |
| 缺陷修复 | `fix/` | `fix/mcscan-reference-validation` |
| 热修复 | `hotfix/` | `hotfix/blast-path-windows` |
| 文档/非代码 | `docs/` | `docs/jcvi-config-guide` |
| 重构 | `refactor/` | `refactor/dispatcher-protocol` |

命名规则：

- 全部小写，单词用 `-` 连接。
- 尽量包含 issue 编号：`feature/42-target-gene-validation`。

## 3. 提交信息规范

采用 [Conventional Commits](https://www.conventionalcommits.org/)。

```text
<type>(<scope>): <简短描述>

<可选：详细说明>

<可选：关联 issue / 破坏性变更说明>
```

常用类型：

- `feat`：新功能
- `fix`：修复
- `docs`：文档
- `style`：不影响代码逻辑的格式调整
- `refactor`：重构
- `test`：测试
- `chore`：构建、依赖、CI 等杂项

示例：

```text
feat(local_synteny): validate target genes against reference BED

- Add _validate_target_genes_in_reference before engine dispatch
- Improve error messages with query/subject labels

Closes #12
```

## 4. 开发流程

### 4.1 开始新任务

```bash
# 1. 同步本地 main
git checkout main
git pull origin main

# 2. 切出功能分支
git checkout -b feature/xxx

# 3. 开发、提交
git add ...
git commit -m "feat(scope): ..."

# 4. 推送到个人 fork 或本仓库
git push -u origin feature/xxx
```

### 4.2 创建 PR

1. 在 GitHub 上从 `feature/xxx` 向 `main` 发起 PR。
2. PR 标题与提交信息格式一致：`feat(scope): 简短描述`。
3. PR 描述中说明：
   - 改动动机
   - 主要变更点
   - 如何验证 / 测试命令
   - 关联 issue（如有）

### 4.3 代码审查

- **核心开发者**：至少 1 人 approval 后方可合并。
- **外部贡献者**：至少 1 名核心开发者 approval。
- 审查关注点：
  - 正确性、边界条件、测试覆盖
  - 是否破坏 CLI / 插件 / 引擎契约
  - 文档是否同步更新
  - 代码风格（由 CI 自动兜底）

### 4.4 合并

- 优先使用 **Squash and merge**，保持 `main` 提交历史线性清晰。
- 合并后删除远程功能分支。

## 5. `main` 分支保护规则

在 GitHub 仓库设置中开启：

1. **Restrict pushes that create files**（可选）
2. **Require a pull request before merging**
   - Require approvals: 1
   - Dismiss stale PR approvals when new commits are pushed
3. **Require status checks to pass before merging**
   - `ruff` / lint
   - `pytest`（platform / engines / integrations）
   - 可选：`pyright` type check
4. **Require branches to be up to date before merging**
5. **Require conversation resolution before merging**
6. **Do not allow bypassing the above settings**（管理员也遵守）
7. **Restrict who can push to matching branches**
   - 仅允许通过 PR 合并，禁止直接 push

## 6. CI / 自动化检查

当前项目使用 `ruff` 与 `pytest`。建议在 `.github/workflows/` 中配置：

```yaml
# .github/workflows/ci.yml 示例
name: CI
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/ruff-action@v3
  test:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: conda-incubator/setup-miniconda@v3
        with:
          environment-file: environment.yml
          activate-environment: genomelens
      - run: python -m pytest platform/tests engines/jcvi/tests integrations/haiant_plugin/tests
```

> 注：若引擎测试依赖 BLAST+ 等外部工具链，可将重测试放到每周或手动触发工作流。

## 7. 版本发布

- `main` 上的提交通过标签触发发布：

```bash
git checkout main
git pull origin main
git tag -a v1.1.0 -m "release: v1.1.0"
git push origin v1.1.0
```

- 版本号遵循 [Semantic Versioning](https://semver.org/lang/zh-CN/)：
  - `MAJOR`：不兼容的 CLI / 插件契约变更
  - `MINOR`：向下兼容的功能新增
  - `PATCH`：向下兼容的问题修复

## 8. 外部贡献者指南

1. Fork 本仓库到自己的账号。
2. 在 fork 中切出 `feature/xxx` 分支。
3. 确保 PR 只包含一个完整改动，避免超大 PR。
4. 首次贡献前阅读：
   - `docs/开发手册/README.md`
   - `docs/开发手册/开发规范.md`
   - `docs/开发手册/开发规范.md`（第 4 节“代码风格”）
   - 本文件
5. 在 PR 中签署 [DCO](https://developercertificate.org/)（如项目需要）：

```text
Signed-off-by: Your Name <email@example.com>
```

## 9. 冲突解决与紧急修复

### 常规冲突

```bash
git checkout feature/xxx
git fetch origin
git rebase origin/main
# 解决冲突后继续
git rebase --continue
git push --force-with-lease origin feature/xxx
```

### 紧急热修复

- 从 `main` 切出 `hotfix/xxx`。
- 修复后走快速 PR / 审查流程。
- 合并后同步到 `main`，并视情况打补丁标签 `vX.Y.Z+1`。

## 10. 快速检查清单

提交 PR 前确认：

- [ ] 基于最新的 `main` 分支
- [ ] 提交信息符合 Conventional Commits
- [ ] `ruff check .` 与 `ruff format --check .` 通过
- [ ] 相关单元测试通过
- [ ] 文档已同步更新（用户文档或开发手册）
- [ ] PR 描述清晰、可验证

---

本方案随项目规模扩大可逐步演进为更严格的 Git Flow 或 Trunk-Based Development。当前阶段以保持简单、保护 `main` 为核心原则。
