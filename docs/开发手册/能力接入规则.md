# 能力接入规则

## probe(探测) 契约

`capabilities` 只表示 engine(引擎) 当前真实可调度的 workflow(工作流)。上游 JCVI 已随包但尚未暴露为工作流的模块，应写入 `bundled_jcvi_modules`，不能混入 `capabilities`。

新增 workflow(工作流) 时必须同步：

- engine dispatcher(分发器)。
- engine probe(探测)。
- shell workflow(外壳工作流) 校验。
- HAIant plugin(智然体插件) 参数选项。
- 最小样例与集成测试。
- 用户文档与开发文档。

## 当前图形能力

默认 `graphics_synteny` 必须输出 dotplot(点图) 与 synteny figure(共线性图)。`graphics_dotplot` 和 `graphics_karyotype` 作为独立图形 workflow(工作流) 暴露，方便用户只生成某一类图件。

## 多物种能力接入要求

当前 shell 已经有 `species[]` 入口，并能将 3 个以上物种拆成 all-vs-all pairwise(全组合两两比较) 子任务。后续继续扩展多物种能力时，不应恢复旧的 `query/subject` CLI 参数，而应沿着 `species[]` 协议继续推进：

- shell 层继续以 `species[]` 作为公开输入模型。
- engine 层 pairwise worker(两两比较工作单元) 可以保留 query/subject manifest(清单)，但不要把它重新暴露为用户入口。
- 调度器应记录 pairwise 组合矩阵和每个子任务状态。
- 中间结果按物种对归档并汇总。
- 图形层后续应生成全局 seqids/layout。
- summary(摘要) 应明确记录每个物种、每个物种对、每个图件和最终总图状态。

文档可以承诺“2 到 n 个物种 pairwise 编排已支持”，但不能把全局 layout 自动优化、最终全局美化总图或机器学习评分写成当前已支持。
