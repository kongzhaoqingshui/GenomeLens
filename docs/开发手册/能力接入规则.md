# 能力接入规则

## 能力分层

GenomeLens 1.X 把能力分成两类：

1. **一站式工作流（One-Stop Workflow）**：面向常见端到端场景，拥有专用 runner 和优化说明，**不能简单等价为一组子模块的线性拼接**。
2. **可编排子模块（Composable Sub-Module）**：单个能力单元，通过显式输入/输出端口组合，供脚本、插件、GUI 使用。

任何新能力接入前，先决定它属于哪一类；若两者都需要，则分别注册。

## probe(探测) 契约

引擎 `probe` 返回当前真实可调度的 workflow 列表。已随包但尚未暴露的 JCVI 模块写入 `bundled_jcvi_modules`，不能混入 `capabilities`。

平台层 `workflow list --json` 把 `one_stop_workflows` 与 `submodules` 合并输出，供 CLI/GUI 消费。

## 新增一站式工作流 checklist

- 在 `platform/src/genomelens/workflow/onestop_registry.py` 注册 `OneStopWorkflowSpec`。
- 指定 `runner`（已有或新建），在 runner 中声明 `ONE_STOP_WORKFLOW_ID`。
- 在 `platform/src/genomelens/cli/commands/analyze.py` 的 `analyze workflow` 中支持该 workflow_id，或复用通用路径。
- 在引擎 dispatcher 中保证对应 `engine_workflow` 可调度。
- 更新 `docs/项目介绍.md`、`docs/用户手册.md` 和根 `README.md`。
- 补充单元/集成测试与最小样例。

## 新增子模块 checklist

- 在 `platform/src/genomelens/workflow/submodule_registry.py` 注册 `SubModuleSpec`，显式声明输入/输出端口、参数、是否可独立运行。
- 在 `platform/src/genomelens/workflow/port_system.py` 中确认所需 `port_kind` 已定义。
- 在 `engines/jcvi/src/jcvi_genomelens/workflow_dispatcher.py` 的 `SUBMODULE_ID_TO_WORKFLOW` 中映射 `module_id -> engine_workflow`。
- 在 `platform/src/genomelens/app/controller/runners/submodule_runner.py` 中添加分支或通用处理。
- 更新 `docs/使用方法/子模块手册.md`。
- 补充端口验证测试与最小样例。

## 图形能力约定

- 默认 `graphics_synteny` 一站式工作流同时输出 dotplot 与 synteny figure。
- `graphics_dotplot`、`graphics_karyotype`、`graphics_histogram`、`graphics_heatmap` 作为独立图形子模块暴露。
- 图形参数统一支持 `figsize`、`dpi`、`formats`。

## 多物种能力接入要求

- shell 层继续以 `species[]` / 自动目录作为公开输入模型。
- engine 层 pairwise worker 保留 query/subject manifest，但不重新暴露为用户入口。
- 调度器记录 pairwise 组合矩阵和每个子任务状态。
- 中间结果按物种对归档并汇总。
- summary 明确记录每个物种、每个物种对、每个图件和最终总图状态。

文档可以承诺“2 到 n 个物种 pairwise 编排已支持”，但不能把全局 layout 自动优化、最终全局美化总图或机器学习评分写成当前已支持。

## 文档与代码同步

新增能力必须同步更新：

- 注册表元数据（端口、参数、默认值、说明）。
- CLI 帮助与示例。
- `docs/项目介绍.md` 能力列表。
- 根 `README.md` 快速开始（若该能力属于常见场景）。
- 对应用户手册或子模块手册。
- 最小集成测试。
