# 能力接入规则

本文约束新增 workflow、submodule、engine 能力时必须同步修改的代码、测试和文档。当前平台协议是：

```text
WorkflowRequest(synteny) -> ExecutionPlan -> PlanOptimizer -> engine manifest v3
SubmoduleRequest(<module_id>) -> ExecutionPlan -> PlanOptimizer -> engine manifest v3
```

层级职责必须保持为：

```text
GUI / CLI / Plugin / Agent
  -> WorkflowRequest 或 SubmoduleRequest

Platform
  -> 校验、规划、预处理、执行 DAG、归档、summary

JCVI Engine
  -> 具体 JCVI 命令、manifest、绘图、engine summary
```

旧 `AnalysisRequest`、`method_config`、`task_kind`、`one_stop_workflow_id`、`sub_module_id`、`port_bindings`、`composition` 不再作为公开请求协议使用。

## 能力分层

GenomeLens 当前把能力分为两类：

1. **一站式工作流**：面向常见端到端场景，当前仅 `synteny`。新增一站式工作流需有明确的端到端优化价值，不能只是子模块的简单拼接。
2. **可编排子模块**：面向脚本、GUI 或插件组合的原子能力，例如 pairwise MCscan、dotplot、karyotype、histogram。

同一个底层 engine workflow 可以同时被一站式工作流和子模块复用，但平台侧必须分别声明其用户入口、端口、参数和产物。

在子模块内部，还必须继续声明 `module_kind`：

- `lightweight`：输入是单一任务域内的原始数据或轻量中间产物，不要求调用方先构造跨 pair / 跨物种聚合输入。
- `aggregate`：输入是 `tracks`、`edges`、聚合 `blocks`、merged BED 等构造式聚合输入，用于总图或汇总结果。

## 平台协议约束

- 公开入口统一使用两类请求协议：
  - `WorkflowRequest` 仅用于一站式 `synteny`。
  - `SubmoduleRequest` 用于所有子模块。
- 物种输入统一使用 `species[]` 和 `reference_index`。
- `query/subject` 只允许作为 engine 内部 pairwise 局部变量或 loader 映射对象，不得重新暴露为公开请求字段或 manifest 顶层字段。
- `WorkflowPlanner` 负责把 `WorkflowRequest` 展开为原始 `ExecutionPlan`。
- `SubmoduleDispatcher` 负责把 `SubmoduleRequest` 包装为单步 `ExecutionPlan`。
- `PlanOptimizer` 负责在 planning 与 execution 之间做去重、共享资源和 artifact 复用等无损重写。
- `PlanExecutor` 是执行 DAG 的唯一入口，消费优化后的 plan，具体步骤委托给 `analysis/execution/handlers/`。
- JCVI manifest 使用 `schema_version=3`，顶层字段为 `workflow`、`task`、`inputs`、`parameters`、`toolchain`、`expected_outputs`、`meta`。
- RunSummary 使用 schema v3，方法扩展放入 `extensions`，子任务放入 `child_runs`，产物放入 `artifact_index`。

## 新增一站式工作流 checklist

1. 在 `platform/src/genomelens/analysis/workflows/onestop.py` 注册 `OneStopWorkflowSpec`。
2. 在 `platform/src/genomelens/analysis/workflows/builtin_mcscan.py` 或对应插件中暴露 workflow 元数据。
3. 在 `platform/src/genomelens/analysis/planning/planner.py` 增加请求展开规则。
4. 在 `platform/src/genomelens/analysis/execution/handlers/` 增加或复用 step handler。
5. 在 `platform/src/genomelens/engines/jcvi/manifest_builder.py` 或对应 engine builder 中生成 manifest。
6. 在 `engines/jcvi/src/jcvi_genomelens/workflows/dispatcher.py` 注册 engine workflow。
7. 在 `engines/jcvi/src/jcvi_genomelens/probe.py` 确认 capability 是否应该对外暴露。
8. 更新 `README.md`、`docs/用户手册.md`、`docs/项目介绍.md` 和相关 `docs/使用方法/` 文档。
9. 补充 planner、manifest builder、executor 和最小 CLI/engine 集成测试。

## 新增可编排子模块 checklist

1. 在 `platform/src/genomelens/analysis/workflows/submodules.py` 注册 `SubModuleSpec`。
2. 为 `SubModuleSpec` 明确声明 `module_kind="lightweight"` 或 `module_kind="aggregate"`。
3. 在 `platform/src/genomelens/analysis/workflows/input_bindings.py` 确认所需端口类型和绑定规则。
4. 在 `platform/src/genomelens/analysis/execution/submodule_mapping.py` 增加 `module_id` 到内部执行请求的映射；必要时在 `platform/src/genomelens/analysis/execution/workflow_mapping.py` 复用同一映射。
5. 在 `analysis/execution/handlers/` 增加或复用 handler。
6. 在 engine dispatcher 注册底层 workflow。
7. 更新 `docs/使用方法/子模块手册.md` 和 `docs/使用方法/SubmoduleRequest JSON.md`。
8. 如果能力需要 HAIant 暴露，同步放入 `integrations/haiant_plugin/` 的 `lightweight/` 或 `aggregate/` 目录，并更新 `PARAMETER_MAPPING.md`。
9. 补充端口校验、submodule mapping 和最小 CLI 集成测试。

## JCVI engine workflow checklist

1. manifest 输入模型位于 `engines/jcvi/src/jcvi_genomelens/manifest/`。
2. runtime 入口位于 `engines/jcvi/src/jcvi_genomelens/runtime/`。
3. workflow 实现按职责放入：
   - `workflows/pairwise/`
   - `workflows/graphics/`
   - `workflows/local_synteny/`
4. 图形底层实现放入 `graphics/`，不要塞回 workflow 根目录。
5. 新 workflow 必须在 `workflows/dispatcher.py` 注册。
6. 若能力应对平台开放，在 `probe.py` 的 capability 列表中声明。
7. 每个 workflow 至少有一个 manifest loader/dispatcher 或 engine integration 测试。

## 图形能力约定

- 图形参数统一优先支持 `figsize`、`dpi`、`formats`。
- `graphics_synteny` 一站式工作流可以复用 pairwise 结果并输出 synteny/dotplot。
- `graphics_dotplot`、`graphics_karyotype`、`graphics_histogram`、`graphics_heatmap` 作为独立子模块暴露；`graphics_histogram` / `graphics_heatmap` 不再作为 workflow_id。
- 原生 local synteny renderer 位于 `engines/jcvi/src/jcvi_genomelens/graphics/local_synteny/`，对 workflow 暴露 `render_local_synteny(...)`。

## 多物种能力约定

- 平台公开请求只表达 `species[]`，多物种配对策略由 planner 决定。
- 多物种无目标基因时，默认展开为 all-vs-all pairwise steps，再聚合全局图。
- 多物种有目标基因时，默认展开为 reference-vs-targets pairwise steps，再聚合 multi local synteny。
- 顶层 summary 必须记录 `child_runs`，不能把 pairwise 路径散落在扁平字段里。

## 文档同步要求

新增能力必须同步更新：

- 用户入口：`README.md`、`docs/用户手册.md`
- 参数与配置：`docs/使用方法/WorkflowRequest JSON.md`、`docs/使用方法/SubmoduleRequest JSON.md`、`docs/使用方法/配置文件说明.md`
- 子模块：`docs/使用方法/子模块手册.md`
- 架构或协议：`docs/开发手册/架构调整/`
- 更新计划或更新日志：`docs/更新计划/`

历史文档只归档到 `docs/历史/`，不得作为当前能力说明继续从主索引暴露。

## 复用扩展约定

- 平台在同一次 `ExecutionPlan` 中存在多个同源 pairwise 子任务时，应优先通过共享执行上下文复用：
  - 已标准化的物种输入目录
  - toolchain 解析结果
  - engine capability probe 结果
- 推荐在 `analysis/execution/` 下通过集中式上下文对象承载上述复用状态，避免在单个 handler 内重复探测、重复准备输入。
- 子模块如果消费上游 pairwise 结果，应优先通过端口绑定传递标准化产物，再由 `submodule_mapping.py` 映射为类型化请求字段；不要把路径字符串散落到各 workflow 私有参数中。
- JCVI manifest 如需复用上游 pairwise 产物，应放入 `inputs.pairwise_artifacts`，由 engine loader 统一校验，不新增并行的顶层私有字段。
- engine workflow 在可满足产物完整性的前提下，应优先复用传入的 `anchors`、`blocks`、`simple`、`layout`、`merged_bed` 等中间结果；仅在缺少必要输入时回退到完整 pairwise 计算。
- 复用属于性能优化，不得改变既有公开 CLI、manifest schema 和 summary schema 的兼容性；新增扩展信息应继续通过 `extensions`、`child_runs`、`artifact_index` 等既有扩展位承载。
