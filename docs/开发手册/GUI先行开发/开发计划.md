# GUI 先行开发计划：JCVI meow / WorkflowRequest V3 + SubmoduleRequest V3

> 本计划取代早期以 `AnalysisRequest v1` 为中心的 GUI 计划。
> 当前平台核心已经收束为两类公开请求协议：`WorkflowRequest(synteny)` 与 `SubmoduleRequest(<module_id>)`，GUI 下一阶段的首要任务不是继续堆页面，而是把请求、结果、运行事件与平台 V3 协议重新对齐。

## 1. 当前状态

### 1.1 已具备能力

- `gui/tauri/` 已有 Tauri v2 + React 18 + TypeScript + Tailwind 桌面应用骨架。
- JCVI meow 品牌、首页工作台、能力入口、设置页、运行页、结果页、图件预览组件已有第一版。
- Rust command 层已具备以下可复用入口：
  - `get_version`
  - `get_template`
  - `get_analysis_schema`
  - `check_environment`
  - `read_request_preview`
  - `run_analysis`
  - `cancel_run`
  - `read_summary`
  - `read_run_log`
  - `read_run_snapshot`
  - `list_artifacts`
  - `open_path`
  - `list_projects`
  - `create_project`
- 运行层已经采用 `genomelens analyze run <request.json>` 子进程模式，符合“GUI 是薄外壳”的方向。
- GUI 构建脚本 `scripts/build_gui.ps1` 已能串联前端 lint/typecheck/test/build 与 Tauri Rust 检查。

### 1.2 当前主要偏差

- TypeScript 请求模型仍以 `AnalysisRequest schema_version=1` 或 `WorkflowRequest v2` 为中心。
- 表单草案、校验、demo request、请求预览文案仍使用旧的 `method / method_config / task_kind` 口径，或仍把 `graphics_histogram`、`graphics_heatmap`、`local_synteny` 当作 workflow_id。
- `RunSummary` 类型仍有较多 `method_data` 扁平字段消费逻辑。
- GUI 本地文档仍把 `AnalysisRequest` 或 `WorkflowRequest v2` 描述为当前协议。
- 当前 GUI 代码尚未完成对 `WorkflowRequest v3`、`SubmoduleRequest v3`、`RunSummary v3`、`artifact_index`、`child_runs`、`extensions` 的完整消费。

## 2. 设计边界

GUI 层负责：

- 项目浏览与任务创建。
- 参数表单、模板读取、字段级提示。
- 本地运行、取消、进度、日志、结果浏览。
- 图件预览、文件打开、环境诊断。
- Agent 对话入口和后续辅助工作台。

GUI 层不负责：

- 实现分析算法。
- 重新编排平台 workflow。
- 私有维护一套与 CLI 不一致的任务协议。
- 在前端复制平台校验和执行 DAG 逻辑。

所有正式分析仍以平台 CLI 为唯一执行入口：

```text
React UI
  -> Tauri Command
  -> genomelens analyze run <request.json>
  -> run_summary.json / run.log / artifacts
```

## 3. 新协议目标

### 3.1 请求协议

GUI 生成和导入的当前任务请求统一为 V3：

- 一站式能力：`WorkflowRequest`
  - `schema_version = 3`
  - `kind = "workflow_request"`
  - `workflow_id = "synteny"`
  - `species[]`、`reference_index`、`parameters`、`output`、`runtime`
- 原子子模块能力：`SubmoduleRequest`
  - `schema_version = 3`
  - `kind = "submodule_request"`
  - `module_id`，例如 `jcvi.graphics_histogram`
  - `inputs`、`parameters`、`output`、`runtime`

GUI 不再把以下字段作为当前协议：

- `method`
- `method_config`
- `task_kind`
- `one_stop_workflow_id`
- `sub_module_id`
- `port_bindings`
- `composition`

这些旧字段只允许出现在历史文档、拒绝旧请求的测试、或迁移说明中。

### 3.2 结果协议

GUI 读取 `RunSummary v3`：

- `schema_version = 3`
- `status`
- `workflow`
- `task`
- `species`
- `final_figures`
- `artifact_index`
- `logs`
- `ui`
- `scoring`
- `extensions`
- `child_runs`

GUI 结果页应优先消费：

- `artifact_index`：图件、中间文件、日志、报告的统一索引。
- `child_runs`：pairwise 子任务或聚合子任务的来源信息。
- `extensions`：workflow 特有补充信息。
- `ui`：进度、主图、摘要路径、日志路径等界面友好字段。

不再新增依赖 `method_data` 扁平字段的界面逻辑。

## 4. 阶段计划

### Phase A：协议重对齐

目标：让 GUI 类型、demo、校验、文档都以 V3 两类请求协议为中心。

- [ ] 替换 `gui/tauri/src/models/analysis-request.ts` 为 `workflow-request.ts` 与 `submodule-request.ts`。
- [ ] 将 `AnalysisRequestDraft` 收束为 `WorkflowRequestDraft` 和 `SubmoduleRequestDraft`，GUI 内部可继续用 `camelCase`，边界输出必须是平台 `snake_case` JSON。
- [ ] 删除或重命名旧 `method_config` 表单聚合逻辑，改为 `parameters.synteny / local_synteny / plot` 或子模块 `parameters`。
- [ ] 更新 `validation.ts`，拒绝旧请求字段，要求 `schema_version=3` 和对应 `kind`。
- [ ] 更新 `gui/demo-data/requests/` 中所有 request JSON。
- [ ] 更新 `read_request_preview` 的 UI 文案和 view model，展示 `kind`、`workflow_id` / `module_id`、物种数、参考物种，而不是 `method`。
- [ ] 保留 Rust command 名称的最小兼容空间，但文档和前端命名应转向 `workflow` / `submodule`。

验收：

- GUI 当前文档不再把 `AnalysisRequest` 或 `WorkflowRequest v2` 当作现行协议。
- GUI 能生成可被 `genomelens analyze schema` 接受的请求。
- demo request 均为 V3。

### Phase B：V3 运行链路恢复

目标：从 GUI 发起一次真实 V3 workflow 或 submodule，并稳定读取日志和终态。

- [ ] `get_template` 调用值切换为 `(kind, id)`：`("workflow", "synteny")` 或 `("submodule", "jcvi.graphics_histogram")`。
- [ ] `run_analysis` 继续保持 `{ requestPath, outdir }`，但前端只传入 V3 request。
- [ ] `analysis:*` 事件携带 `runId / outdir / requestPath / logPath / summaryPath`。
- [ ] 进度来源优先使用 `run.log` 中稳定事件，不解析易变 stdout 文本。
- [ ] 失败页展示 `run.log` 尾部、退出码、summary 路径和可打开的输出目录。

验收：

- GUI 可运行 `app/onestop/gljcvi-synteny` 或等价 demo 数据生成真实结果。
- 运行中页面不会因 stdout 格式变化失效。
- 取消运行、失败运行、成功运行都有终态事件。

### Phase C：结果页升级到 RunSummary v3

目标：结果浏览不再依赖旧扁平字段。

- [ ] 重写 `run-summary.ts`，镜像 `platform/src/genomelens/contracts/summaries.py` 的 v3 稳定字段。
- [ ] `run-summary-view.ts` 优先从 `artifact_index` 构建图件列表。
- [ ] pairwise 子任务视图改读 `child_runs`。
- [ ] 多物种、local synteny、histogram、heatmap 的特有字段改读 `extensions`。
- [ ] `list_artifacts` 与结果页都尊重 `ArtifactRecord.owner_task / input_refs / provenance`。
- [ ] 空结果、部分失败、仅日志存在、summary 缺失等状态都有可读提示。

验收：

- 双物种 synteny、多物种 global karyotype、multi local synteny、histogram、heatmap 至少各有一个可浏览结果形态。
- 结果页主入口来自 `ui.primary_figures` 或 `artifact_index.preview`。

### Phase D：工作台表单重建

目标：把“一个大表单”拆成两类能力入口：一站式 workflow 和子模块。

- [ ] `synteny` 一站式：双物种与多物种共线性入口，支持 reference 选择、目标基因局部共线性。
- [ ] 子模块工作台：按 `lightweight` / `aggregate` 分组，选择 module_id 后动态渲染端口和参数表单。
- [ ] 常用子模块快捷入口：`jcvi.graphics_histogram`、`jcvi.graphics_heatmap`、`jcvi.local_synteny`、`jcvi.graphics_dotplot`。
- [ ] 每个表单只暴露平台支持的参数，不在 GUI 中发明隐藏参数。
- [ ] 表单的“高级参数”直接映射到对应分组参数或 `parameters.extras`。

验收：

- 每个表单都能生成可读、可保存、可复跑的 V3 请求。
- 参数默认值来自 `analyze template <kind> <id>`，不是前端硬编码。

### Phase E：项目与工作区体验

目标：把当前项目列表与运行输出管理打磨成稳定的桌面体验。

- [ ] 明确 workspace/project/outdir 的目录约定。
- [ ] `create_project` 创建最小项目元数据。
- [ ] 最近项目、最近运行、输出目录恢复读取。
- [ ] 运行页可从历史 outdir 恢复 `read_run_snapshot`。
- [ ] 文件选择、路径校验、打开目录在 Windows/macOS 都有一致反馈。

验收：

- 关闭 GUI 后重新打开，能继续查看最近运行结果。
- project 不承载平台业务状态，只保存 GUI 本地索引和偏好。

### Phase F：质量与发布预备

目标：让 GUI 进入可持续集成前的“本地可验收”状态。

- [ ] 前端：`pnpm lint`、`pnpm typecheck`、`pnpm test`、`pnpm build`。
- [ ] Rust：`cargo check`、`cargo clippy -- -D warnings`。
- [ ] 平台联动：至少一条真实 workflow 和一条真实 submodule 从 GUI 跑通。
- [ ] 文档同步：`gui/README.md`、`gui/tauri/README.md`、`gui/demo-data/README.md`、`docs/开发手册/GUI先行开发/`。
- [ ] 构建脚本输出位置与版本号说明更新。

验收：

- `scripts/build_gui.ps1` 可作为本地 GUI 验证主入口。
- debug 构建产物能运行当前平台 V3 请求。

## 5. 三人分工建议

### A：Tauri / CLI 集成

- 保持 `run_analysis`、`cancel_run`、日志读取、summary 读取稳定。
- 将 `get_template(method)` 的实现语义迁移为 `(kind, id)` 模板读取；后续可重命名入参。
- 更新 Rust command 的错误信息，明确 V3 request 要求。
- 维护 Windows/macOS 路径、可执行文件发现、conda 环境诊断。

### B：界面 / 交互

- 重建 workflow 与子模块选择表单页面。
- 将结果页切到 `artifact_index + child_runs + extensions`。
- 优化空状态、失败态、运行态和图件预览。
- 保持 JCVI meow 的视觉风格，但避免在数据图页面加入干扰科研阅读的装饰。

### C：模型 / 数据流

- 负责 TypeScript request、summary、artifact、validation、view-model 的 V3 迁移。
- 维护 demo request 和端到端烟测数据。
- 确保 GUI draft 与平台 JSON 的双向映射可测。
- 维护本目录数据契约文档。

## 6. 禁止事项

- 不再新增基于 `AnalysisRequest v1` 或 `WorkflowRequest v2` 的 GUI 功能。
- 不在前端复制 `WorkflowPlanner` 或 engine manifest 构造逻辑。
- 不让 GUI 直接生成 engine manifest。
- 不把 GUI 本地 `camelCase` draft 泄漏为平台请求文件。
- 不在结果页硬编码某一种 workflow 的路径推断，优先读 `artifact_index`。
