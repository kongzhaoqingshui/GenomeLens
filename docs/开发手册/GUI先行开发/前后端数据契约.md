# GUI 前后端数据契约草案

> 本文件记录 GUI Phase 0 的前后端通信契约草案。后续 B/C 在表单、状态机和结果解析上扩展时，应先更新本文件再落实现。

---

## 1. 通信原则

- GUI 前端通过 Tauri Command 调用 Rust 后端。
- Rust 后端通过子进程或 sidecar 调用 `genomelens` / `jcvi-genomelens`。
- GUI 不直接实现分析业务逻辑，不维护与 CLI 不一致的私有任务协议。
- 跨层数据使用可序列化 JSON 结构。
- 对于 `get_template()`、`get_analysis_schema()`、`check_environment()`、`read_summary()` 这类直接透传 platform CLI / JSON 文件的接口，**返回载荷保持平台既有字段命名**，不在 Tauri/Rust 层改写键名；当前 Phase 0/1 实际上以 `snake_case` 为准。
- GUI 本地表单状态、store、view model 可以使用 `camelCase`，但必须在边界层完成与平台 JSON 的双向转换。

---

## 2. Phase 0 Command

### `get_version()`

用于探测本地 platform CLI 与 JCVI engine 可用性。

```typescript
interface CommandVersion {
  ok: boolean;
  command: string;
  version: string;
  error?: string;
}

interface VersionInfo {
  platform: CommandVersion;
  engine: CommandVersion;
}
```

语义：

- `platform` 调用 `genomelens --version`。
- `engine` 调用 `jcvi-genomelens probe`。
- 当命令不存在或退出码非 0 时，`ok = false`，错误摘要写入 `error`。
- 该命令只做能力探测，不修改任何本地状态。

### `get_template(method)`

用于把平台侧 request 模板暴露给 GUI 表单。

```typescript
type GetTemplateInput = { method: "mcscan" | string };
type GetTemplateOutput = AnalysisRequest;
```

语义：

- Rust 后端调用 `genomelens analyze template <method>`。
- 当前先行版默认使用 `method = "mcscan"`。
- 返回值必须是平台 CLI 输出的 JSON，不在 GUI 后端重写模板默认值，也不改写原始 `snake_case` 字段名。

### `get_analysis_schema()`

用于把平台侧 AnalysisRequest JSON schema 暴露给 GUI 表单与验证层。

```typescript
type GetAnalysisSchemaOutput = Record<string, unknown>;
```

语义：

- Rust 后端调用 `genomelens analyze schema`。
- 返回值直接来自平台 CLI 的 JSON schema。
- GUI 可基于该 schema 做字段说明和动态表单校验，但不得修改平台 schema 语义。
- 若 GUI 侧需要 `camelCase` draft model，应在前端边界层自行映射，不额外派生第二套 platform schema。

## 4. 事件流草案

```typescript
type GuiEvent =
  | { name: "analysis:stdout"; payload: { runId: string; outdir: string; requestPath: string; startedAt?: string; line: string } }
  | { name: "analysis:state"; payload: { runId: string; outdir: string; requestPath: string; startedAt?: string; state: string; progress: number } }
  | { name: "analysis:finished"; payload: { runId: string; outdir: string; requestPath: string; startedAt?: string; finishedAt?: string; exitCode?: number | null; logPath?: string; summaryPath?: string; status: "SUCCEEDED" | "FAILED" | "CANCELLED"; summary?: RunSummary } }
  | { name: "analysis:error"; payload: { runId: string; outdir: string; requestPath: string; startedAt?: string; finishedAt?: string; exitCode?: number | null; logPath?: string; summaryPath?: string; message: string; code?: string; details?: Record<string, unknown> } };
```

后续状态机由成员 C 主导映射，成员 A 提供 Rust 事件推送能力，成员 B 负责展示组件。

Phase 2 当前验证结论：

- 终端 stdout/stderr 主要承载 CLI 展示文本与进度条，不适合作为稳定契约直接解析。
- `outdir/logs/run.log` 中的 `task_started` / `task_finished` 行包含稳定的 `step=` / `status=` 信息，可作为 `analysis:state` 的第一版来源。
- `analysis:stdout` 在 Phase 2 当前承载的是“稳定的运行日志行流”，实现上允许优先复用 `run.log` 的新增行，而不是强依赖子进程终端输出。
- `analysis:stdout` 目前**不是**严格意义上的子进程原始终端 stdout；若后续需要区分终端输出与日志流，再拆分为独立事件。
- Phase 2 起，四类分析事件都稳定携带 `runId` / `outdir` / `requestPath` 作为关联上下文字段，GUI 不需要再从本地运行态或 `details` 里反推事件归属。
- `analysis:finished` / `analysis:error` 可额外携带 `finishedAt`，供 GUI 在运行生命周期面板中展示结束时间或判断终态刷新时机。
- `RunHandle`、`analysis:finished`、`analysis:error` 可稳定携带 `logPath` / `summaryPath`，供 GUI 直接跳转日志与结果文件，不必再从 `outdir` 约定推导。
- `analysis:finished` / `analysis:error` 可稳定携带 `exitCode`；其中正常成功通常为 `0`，非零退出保留平台子进程原始退出码，异常终止时允许为 `null`。
- 对于已经成功 spawn 的 run，后端会在终态路径上保证至少发出一次 `analysis:finished`；`wait_failed` 这类异常等待失败也不应让 GUI 停留在“永远运行中”的生命周期状态。
- 终态收尾时允许把 `run.log` 中最后一条未以换行结尾的稳定日志行作为 `analysis:stdout` 发出，避免尾行在进程退出时丢失。
- multi-species run 的 `analysis:state` 第一版已覆盖稳定 step 映射：`prepare_multi_species_workspace -> PREPARING_WORKSPACE`，`run_pairwise_job -> RUNNING_ENGINE`，`copy_pairwise_figures -> PARSING_ENGINE_SUMMARY`，`optimize_layout` / `build_global_karyotype` / `write_multi_summary -> FINALIZING`。

### 5.4 GUI-owned Tauri command 命名约定

以下命令不是 platform JSON 直通载荷，Tauri command 输入/输出统一使用 `camelCase`：

- `list_projects`
- `create_project`
- `read_request_preview`
- `list_artifacts`
- `run_analysis`
- `read_summary`
- `read_run_log`
- `open_path`

`read_request_preview` 的职责边界：

- 仅用于 GUI 导入已有 request 文件时读取本地 JSON 预览。
- 输入保持 GUI-owned `camelCase`：`{ requestPath }`。
- 输出仅返回 `requestPath`、原始 `json`、以及便于界面分流的 `method?` / `workflow?` 轻量摘要。
- 缺少 `method` / `workflow` 时不报业务错误，返回 `null` 交给 GUI 做轻量提示或后续分流。
- 目录路径、非文件路径、无效 JSON、以及超出 GUI 预览上限的超大文件应直接返回清晰错误；这仍属于本地文件读取边界，不属于 platform request 业务校验。
- 不做业务校验、不改写 platform JSON 字段名、不推导 `outdir`、不替代 `get_template()` / `get_analysis_schema()`。
- 之所以使用 Tauri command 而不是前端直接 `plugin-fs` 读取，是因为当前 `fs:default` 仅覆盖 app-specific dirs，不开放任意用户选中文件路径读取。

`run_analysis` 的本地输入边界：

- `requestPath` 进入子进程前应至少满足“非空、存在、且是常规文件”，失败时返回可直接展示给 GUI 的本地路径错误。
- 该层只做薄校验与错误信息增强，不展开 AnalysisRequest 业务语义校验，也不改变 `{ requestPath, outdir }` 载荷结构。

### 5.5 启动层 ready state（GUI local）

JCVI 喵桌面体验的启动层可以聚合以下 3 个 command 的 warmup 状态，但这只是 GUI 本地 view-model，不改变底层命令契约：

- `get_version()`
- `get_template("mcscan")`
- `get_analysis_schema()`

建议的 GUI local 状态字段：

- `status`: `loading | ready | error`
- `pending`: 仍在加载的资源键，当前固定为 `version | template | schema`
- `failed`: 加载失败的资源键
- `readyCount / totalCount`: 用于 splash 进度展示
- `activeHint / hints`: 启动层轮播提示文案
- `diagnosticsRoute`: 失败或超时后跳转环境诊断页，当前固定为 `/settings`

该层只负责聚合 command 的 loading/error/ready，不改写 `VersionInfo`、`AnalysisRequest` 或 schema 的原始结构。

### 5.6 JCVI 能力入口 view-model（GUI local）

首页能力环和工作台入口可以稳定消费一组 GUI-owned capability entries，不新增 command/event 字段：

- `id`
- `title`
- `subtitle`
- `description`
- `route`
- `status`: `connected | reserved`
- `statusLabel`
- `workflowPreset?`

Phase 2 当前目标 route：

- `Pairwise Synteny` -> `/analysis/new`
- `Multi-species Synteny` -> `/analysis/new`
- `Local Synteny` -> `/analysis/new`
- `Dotplot` -> `/analysis/new`
- `Karyotype` -> `/analysis/new`
- `Ortholog Catalog` -> `/analysis/new`
- `Environment Check` -> `/settings`

其中 `workflowPreset` 仅用于 GUI 本地 draft helper 预设 `mcscan.workflow`，不改变 platform JSON 结构，不新增 `outdir` 或其他传输层字段。
