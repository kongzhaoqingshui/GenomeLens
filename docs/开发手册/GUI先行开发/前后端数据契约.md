# GUI 前后端数据契约：WorkflowRequest V3 / SubmoduleRequest V3 / RunSummary V3

> 本文是 GUI 与平台核心之间的当前契约。旧 `AnalysisRequest v1`、`method_config`、扁平 `method_data` 只作为历史迁移对象存在，不再作为新功能设计依据。

## 1. 通信原则

- GUI 前端通过 Tauri Command 调用 Rust 后端。
- Rust 后端通过子进程或 sidecar 调用 `genomelens` / `jcvi-genomelens`。
- GUI 不直接实现分析业务逻辑，不维护与 CLI 不一致的私有任务协议。
- 平台 JSON 边界保持 `snake_case` 字段名。
- GUI 本地表单、store、view model 可以使用 `camelCase`，但必须在边界层显式转换。
- 正式执行入口固定为 `genomelens analyze run <request.json>`，`request.json` 可以是 `WorkflowRequest` 或 `SubmoduleRequest`。

GUI、平台和 engine 的职责边界如下：

```text
GUI / CLI / Plugin / Agent
  -> WorkflowRequest 或 SubmoduleRequest

Platform
  -> 校验、规划、预处理、执行 DAG、归档、summary

JCVI Engine
  -> 具体 JCVI 命令、manifest、绘图、engine summary
```

GUI 侧不得直接生成 engine manifest，也不得复制 `WorkflowPlanner`、输入预处理、artifact 归档或 RunSummary 汇总逻辑。Rust/Tauri 后端只作为桌面桥接层，负责命令调用、路径访问、进程生命周期、事件流和结果读取。

## 2. 请求模型

### 2.1 WorkflowRequest

```typescript
export type WorkflowRequestKind = "workflow_request";

export interface WorkflowRequest {
  schema_version: 3;
  kind: WorkflowRequestKind;
  workflow_id: "synteny";
  species: WorkflowSpeciesInput[];
  reference_index: number;
  inputs: Record<string, unknown>;
  parameters: WorkflowParameters;
  output: WorkflowOutput;
  runtime: WorkflowRuntime;
}

export interface WorkflowSpeciesInput {
  name: string;
  input_mode: "bed_cds" | "gff_genome";
  bed?: string;
  cds?: string;
  gff?: string;
  genome?: string;
  aliases?: string[];
  metadata?: Record<string, unknown>;
}

export interface WorkflowParameters {
  synteny?: Record<string, unknown>;
  local_synteny?: Record<string, unknown>;
  plot?: Record<string, unknown>;
  histogram?: Record<string, unknown>;
  heatmap?: Record<string, unknown>;
  extras?: Record<string, unknown>;
}

export interface WorkflowOutput {
  directory: string;
  force?: boolean;
  formats?: string[];
}

export interface WorkflowRuntime {
  project_config?: string;
  engine_config?: string;
  jcvi_engine?: string;
  blastn?: string;
  makeblastdb?: string;
  lastal?: string;
  lastdb?: string;
  threads?: number | null;
  min_block_size?: number | null;
  log_level?: string;
  verbose?: boolean;
  console_log?: boolean;
}
```

### 2.2 SubmoduleRequest

```typescript
export type SubmoduleRequestKind = "submodule_request";

export interface SubmoduleRequest {
  schema_version: 3;
  kind: SubmoduleRequestKind;
  module_id: string;
  inputs: Record<string, unknown>;
  parameters: Record<string, unknown>;
  output: WorkflowOutput;
  runtime: WorkflowRuntime;
}
```

### 2.3 GUI 草案模型

GUI 可维护 `WorkflowRequestDraft` 与 `SubmoduleRequestDraft`：

- 字段可用 `camelCase`。
- 可包含 UI-only 字段，例如当前步骤、折叠状态、路径选择状态、临时校验消息。
- 保存为平台请求文件前必须经过 `toWorkflowRequest(draft)` / `toSubmoduleRequest(draft)`。
- 从平台 JSON 导入时必须经过 `fromWorkflowRequest(request)` / `fromSubmoduleRequest(request)`。

禁止把 GUI-only 字段写入正式 request JSON。

### 2.4 旧字段拒绝

GUI 校验层应把以下字段视为旧协议残留：

- `method`
- `method_config`
- `task_kind`
- `one_stop_workflow_id`
- `sub_module_id`
- `port_bindings`
- `composition`

导入包含这些字段的旧请求时，GUI 应提示“这是旧版请求，需要迁移”，而不是静默修补后运行。

## 3. Tauri Command

### 3.1 平台透传类

| Command | 当前实现输入 | 目标语义 | 返回 |
|---|---|---|---|
| `get_version` | 无 | 探测 platform / engine 版本 | `VersionInfo` |
| `get_template` | `{ kind: "workflow" | "submodule"; id: string }` | 读取平台模板 | `WorkflowRequest` / `SubmoduleRequest` |
| `get_analysis_schema` | 无 | 返回当前联合 JSON schema | JSON schema |
| `check_environment` | 无 | 运行环境诊断 | check JSON |

说明：`get_template` 目前 Rust 入参名仍可能叫 `method`，这是实现遗留。前端新代码应把它当作 `(kind, id)` 使用，例如 `(workflow, synteny)`、`(submodule, jcvi.graphics_histogram)`。

### 3.2 GUI-owned 类

| Command | 输入 | 输出 | 说明 |
|---|---|---|---|
| `read_request_preview` | `{ requestPath: string }` | `RequestPreview` | 只读预览本地请求 JSON |
| `run_analysis` | `{ requestPath: string; outdir: string }` | `RunHandle` | 启动分析子进程 |
| `cancel_run` | `{ runId: string }` | `CancelRunResult` | 取消当前 GUI 会话内登记的运行 |
| `read_summary` | `{ outdir: string }` | `RunSummary` | 读取 `run_summary.json` |
| `read_run_snapshot` | `{ outdir: string; tailLines?: number }` | `RunSnapshot` | 聚合 summary、log、artifact |
| `read_run_log` | `{ outdir: string; tailLines?: number }` | `RunLogSnapshot` | 读取 `run.log` |
| `list_artifacts` | `{ outdir: string }` | `ArtifactSummary[]` | 基于 `artifact_index` 和目录扫描列出结果 |
| `open_path` | `{ path: string }` | `void` | 打开文件或目录 |
| `list_projects` | `{ workspace: string }` | `ProjectSummary[]` | GUI 本地项目索引 |
| `create_project` | `{ workspace: string; name: string }` | `ProjectSummary` | 创建 GUI 项目目录 |

GUI-owned command 可以使用 `camelCase` 载荷；平台 JSON 本身不改名。

## 4. 请求预览

目标输出：

```typescript
export interface RequestPreview {
  requestPath: string;
  json: Record<string, unknown>;
  schemaVersion: number | null;
  kind: "workflow_request" | "submodule_request" | string | null;
  workflowId?: string | null;
  moduleId?: string | null;
  speciesCount: number;
  referenceName?: string | null;
}
```

职责边界：

- 只读文件并做轻量结构摘要。
- 不执行平台业务校验。
- 不推导 outdir。
- 不修改 request JSON。
- 文件不存在、非 JSON、超出预览上限时返回清晰本地错误。

## 5. 运行事件

```typescript
type GuiEvent =
  | {
      name: "analysis:stdout";
      payload: {
        runId: string;
        outdir: string;
        requestPath: string;
        startedAt?: string;
        line: string;
      };
    }
  | {
      name: "analysis:state";
      payload: {
        runId: string;
        outdir: string;
        requestPath: string;
        startedAt?: string;
        state: string;
        progress: number;
      };
    }
  | {
      name: "analysis:finished";
      payload: {
        runId: string;
        outdir: string;
        requestPath: string;
        startedAt?: string;
        finishedAt?: string;
        exitCode?: number | null;
        logPath?: string;
        summaryPath?: string;
        status: "SUCCEEDED" | "FAILED" | "CANCELLED";
        summary?: RunSummary;
      };
    }
  | {
      name: "analysis:error";
      payload: {
        runId: string;
        outdir: string;
        requestPath: string;
        startedAt?: string;
        finishedAt?: string;
        exitCode?: number | null;
        logPath?: string;
        summaryPath?: string;
        message: string;
        code?: string;
        details?: Record<string, unknown>;
      };
    };
```

约定：

- `analysis:stdout` 在 GUI 中表示“稳定日志流”，可以来自 `run.log` 新增行，不要求等同子进程原始 stdout。
- `analysis:state` 优先从平台稳定日志事件映射，不从富文本进度条解析。
- 成功 spawn 的 run 必须最终收敛到 `analysis:finished` 或 `analysis:error`。
- 终态事件应尽量携带 `exitCode`、`logPath`、`summaryPath`。

## 6. RunSummary V3

```typescript
export interface RunSummary {
  schema_version: 3;
  status: string;
  workflow?: string | null;
  task?: Record<string, unknown> | null;
  species?: unknown[];
  final_figures?: string[];
  artifact_index?: ArtifactRecord[];
  logs?: Record<string, unknown>;
  ui?: RunSummaryUi;
  scoring?: Record<string, unknown>;
  extensions?: Record<string, unknown>;
  child_runs?: ChildRunRecord[];
}
```

### 6.1 GUI 稳定消费字段

- `status`
- `schema_version`
- `workflow`
- `task`
- `species`
- `final_figures`
- `artifact_index`
- `logs`
- `ui`
- `scoring`
- `extensions`
- `child_runs`

### 6.2 `ui`

GUI 可稳定读取：

- `ui.state`
- `ui.progress`
- `ui.primary_figures`
- `ui.summary_path`
- `ui.log_path`

### 6.3 `artifact_index`

GUI 可稳定读取：

- `artifact_id`
- `artifact_type`
- `path`
- `format`
- `preview`
- `owner_task`
- `input_refs`
- `provenance`
- `metadata`

### 6.4 `child_runs`

GUI 用 `child_runs` 表示 pairwise 子任务、聚合子任务和 plot-only 子任务：

- `run_id` 或 `child_id`
- `workflow`
- `status`
- `outdir`
- `summary_path`
- `artifact_ids`
- `inputs`
- `metadata`

具体字段以平台 `ChildRunRecord` 为准；前端应容忍额外字段。

### 6.5 `extensions`

`extensions` 用于 workflow 特有信息，例如：

- `species_count`
- `pairing_strategy`
- `reference_name`
- `global_figures`
- `multi_species_local_figures`
- `jcvi_backend`
- `jcvi_workflow`
- `native_layout`
- `native_multi_species`

GUI 可以显示这些字段，但不应把它们提升为所有 workflow 都必须存在的顶层契约。

## 7. Artifact 视图

结果页构建顺序：

1. 优先读取 `artifact_index`。
2. 用 `ui.primary_figures` 标记首屏主图。
3. 用 `final_figures` 做旧路径或最终图件兜底。
4. `list_artifacts` 可作为目录扫描补充，不应覆盖 summary 中的 provenance。

图件类型建议：

- `figure`
- `table`
- `log`
- `report`
- `intermediate`
- `request`
- `summary`

## 8. Workflow 表单到请求的映射

### 8.1 `synteny`

- `workflow_id = "synteny"`
- `species[]`：两个或多个物种。
- `reference_index`：参考物种下标。
- `parameters.synteny`：共线性、过滤、all-vs-all、reference-vs-targets 等参数。
- `parameters.local_synteny`：目标基因、窗口、native renderer 等参数。
- `parameters.plot`：karyotype、dotplot、local synteny 等绘图参数。

### 8.2 子模块表单

子模块表单生成 `SubmoduleRequest`：

- `module_id`：例如 `jcvi.graphics_histogram`、`jcvi.graphics_heatmap`、`jcvi.local_synteny`。
- `inputs`：直接端口绑定。
- `parameters`：模块特定参数。
- `output`：输出目录、覆盖策略、格式。

不再通过 `WorkflowRequest.workflow_id` 表达 histogram、heatmap、local_synteny 等原子能力。

## 9. 迁移清单

- [ ] `analysis-request.ts` -> `workflow-request.ts` + `submodule-request.ts`。
- [ ] `analysis-request-draft.ts` -> `workflow-request-draft.ts` + `submodule-request-draft.ts`。
- [ ] `request-preview.ts` 支持 `kind`、`workflowId`、`moduleId` 摘要。
- [ ] `validation.ts` 切到 V3 校验，拒绝旧字段。
- [ ] `run-summary.ts` 切到 v3。
- [ ] `run-summary-view.ts` 从 `artifact_index / child_runs / extensions` 构建视图。
- [ ] `jcvi-meow.ts` 的能力 preset 区分为 workflow（`synteny`）和 submodule（9 个 module_id）。
- [ ] `gui/demo-data/requests/` 全部换成 V3 request。
- [ ] 文档、测试、页面文案不再把 `AnalysisRequest` 或 `WorkflowRequest v2` 当作当前协议。

## 10. 测试要求

- TypeScript request roundtrip：draft -> platform JSON -> draft。
- 旧字段拒绝：含 `method_config` 的 JSON 不能作为当前请求运行。
- `get_template("workflow", "synteny")` 返回 V3 WorkflowRequest。
- `get_template("submodule", "jcvi.graphics_histogram")` 返回 V3 SubmoduleRequest。
- `read_request_preview` 能摘要 V3 WorkflowRequest 与 SubmoduleRequest。
- `run_analysis` 能启动真实 demo。
- `read_summary` 能消费 v3 summary。
- 结果页可从 `artifact_index` 展示图件。
