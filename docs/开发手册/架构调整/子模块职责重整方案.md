# 子模块职责重整方案（全量重整 + 干净切换）

> 本文定义把当前 10 个 haiant_plugin 子模块重整为「真正纯模块化」的目标分层与迁移路径。
> 决策口径：**全量重整**（合并重复、解耦计算与渲染、统一产物词汇与参数模型、onestop 编排子模块）+ **干净切换**（不保留旧 `module_id` / 旧参数名 / 旧端口名的兼容；旧插件 zip 与旧请求 JSON 无需继续可用）。
> 本方案是 `最终架构目标.md` 中「Platform 只编排、不重写绘图」「Artifact 可追踪」「任务协议稳定」三条原则在子模块层的落地。

---

## 1. 问题诊断（职责冲突与重合）

经四层（engine workflow / platform 注册表与分发 / plugin 入口与资产 / 构建脚本）排查，当前 10 子模块的「伪模块化」问题确认如下：

### 1.1 计算与渲染纠缠（最严重）
4 个「可视化」子模块（`graphics_dotplot` / `graphics_synteny` / `graphics_karyotype` / `local_synteny`）声称只画图，但在 engine 侧通过
`ensure_pairwise_artifacts(..., fallback_runner=run_pairwise)` 在缺产物时**偷偷重跑整条 pairwise 计算**。

- `engines/jcvi/.../graphics/synteny.py:25-31`
- `engines/jcvi/.../graphics/dotplot.py:68-73`
- `engines/jcvi/.../graphics/karyotype.py:87-96`
- `engines/jcvi/.../graphics/local_synteny/single.py:247-253`
- 机制源头：`engines/jcvi/.../workflows/pairwise/artifact_reuse.py:18-45`

后果：渲染模块隐式依赖 `species_pair`，与「显式输入、显式输出」的子模块契约相悖；同一份 pairwise 在「先算后画」时被算两遍；用户无法单独复用渲染。

### 1.2 计算模块重复
`mcscan_pairwise` 与 `catalog_ortholog` 走同一条 `build_pairwise_manifest` 路径，
平台注册表参数除 `threads` 外**逐字节相同**（`platform/.../workflows/submodules.py` mcscan 143-204 vs catalog 381-426），
engine 侧 ortholog 实为 `mcscan.run` 在 `align_soft ∈ {last, diamond_blastp}` 时分派 `_run_with_catalog`。二者实质是「同一计算的不同输出选择」，却暴露成两个并列模块。

### 1.3 产物词汇与端口命名不一致
- 端口别名漂移：`layout` vs `karyotype_layout`（`submodule_mapping.py:116-127` 用 `or` 兜底），`use_native_renderer`（WorkflowRequest）vs `use_native_local_synteny_renderer`（SubmoduleRequest）。
- 同一产物在不同模块的端口名、formats 词汇不统一，迫使映射层写补丁。

### 1.4 onestop 与子模块的编排错位
onestop planner 不复用子模块语义，而是在 `handlers/pairwise.py:64-65` 直接**改写 engine_workflow**（target_gene_ids 存在时重写为 `local_synteny`），导致编排逻辑与子模块定义两套真相。

### 1.5 通用绘图与领域绘图混层
`histogram` / `heatmap` 是领域无关的通用绘图，却与共线性领域渲染模块并列在同一「lightweight」类目，类目无法表达「这两个不依赖任何 pairwise 产物」。

---

## 2. 目标分层（4 层纯模块化）

按「产物流向」而非「UI 习惯」重新分层。每个模块只属于一层，层内职责单一：

| 层 | 职责 | 模块（新 module_id） | 输入 | 输出 |
|---|---|---|---|---|
| **compute（计算层）** | 产出数据产物，**绝不渲染** | `jcvi.pairwise` | `species_pair` | anchors / blocks / simple /（可选）ortholog catalog |
| **render（渲染层）** | 纯渲染，**绝不计算 pairwise**，缺产物即报错 | `jcvi.render_dotplot`、`jcvi.render_synteny`、`jcvi.render_karyotype`、`jcvi.render_local_synteny` | 上游数据产物（anchors/blocks/simple/layout/target_genes） | 图件 |
| **aggregate（聚合层）** | 多比较对 / 多物种汇总成总图 | `jcvi.aggregate_global_karyotype`、`jcvi.aggregate_multi_local_synteny` | tracks/edges/多 layout | 总图 |
| **generic（通用层）** | 领域无关绘图 | `jcvi.plot_histogram`、`jcvi.plot_heatmap` | numeric_files / matrix_csv | 图件 |

模块数 10 → **9**（mcscan_pairwise + catalog_ortholog 合并为 `jcvi.pairwise`）。

### 2.1 `jcvi.pairwise` 的 ortholog 输出
ortholog 由参数 `emit_ortholog: bool`（或 `output.formats` 含 `ortholog`）选择，
而非独立模块。engine 侧依旧由 `align_soft` 决定比对后端；ortholog catalog 作为该计算的一项**可选产物**声明在 `expected_outputs`。

## 3. 旧 → 新 清单映射（干净切换，无兼容层）

### 3.1 module_id
| 旧 module_id | 新 module_id | 处置 |
|---|---|---|
| `jcvi.mcscan_pairwise` | `jcvi.pairwise` | 重命名 + 合并 |
| `jcvi.catalog_ortholog` | （并入 `jcvi.pairwise`，`emit_ortholog=true`） | **删除模块/资产/entry** |
| `jcvi.graphics_dotplot` | `jcvi.render_dotplot` | 重命名 + 去计算 |
| `jcvi.graphics_synteny` | `jcvi.render_synteny` | 重命名 + 去计算 |
| `jcvi.graphics_karyotype` | `jcvi.render_karyotype` | 重命名 + 去计算 |
| `jcvi.local_synteny` | `jcvi.render_local_synteny` | 重命名 + 去计算 |
| `jcvi.graphics_histogram` | `jcvi.plot_histogram` | 重命名 + 归通用层 |
| `jcvi.graphics_heatmap` | `jcvi.plot_heatmap` | 重命名 + 归通用层 |
| `jcvi.graphics_karyotype_global` | `jcvi.aggregate_global_karyotype` | 重命名 |
| `jcvi.local_synteny_multi` | `jcvi.aggregate_multi_local_synteny` | 重命名 |

### 3.2 参数 / 端口
- `use_native_renderer` / `use_native_local_synteny_renderer` → 统一 **`use_native_renderer`**（两侧请求模型一致）。
- 端口 `karyotype_layout` → 统一 **`layout`**；移除 `submodule_mapping.py` 的 `or` 兜底。
- histogram 列/分箱参数统一加 `histogram_` 前缀（与 plot_heatmap 对齐命名风格），消除 WorkflowRequest 与 SubmoduleRequest 的不带前缀漂移。

### 3.3 engine workflow 字符串
engine `_WORKFLOW_REGISTRY` 与 `contract.SUPPORTED_WORKFLOWS` 同步重命名；
**不保留** `WORKFLOW_ALIASES` 旧别名（干净切换）。

## 4. 分阶段迁移计划（每阶段 ruff + pytest 全绿后再进下一阶段）

| 阶段 | 内容 | 主要文件 |
|---|---|---|
| **P1 合并计算模块** | mcscan_pairwise + catalog_ortholog → `jcvi.pairwise`（ortholog 转可选输出）；删除 catalog 模块/资产/entry/featureMap 项 | engine: dispatcher/contract/command_mapping/mcscan; platform: submodules/submodule_mapping/manifest_builder; plugin: entry/assets/build script; tests |
| **P2 渲染去计算** | 移除 4 处 `fallback_runner=run_pairwise`；`artifact_reuse` 缺产物改**报错**；渲染模块 module_id 改 `render_*` | engine: graphics/synteny,dotplot,karyotype,local_synteny/single,artifact_reuse; platform: submodules |
| **P3 统一产物词汇与端口** | `karyotype_layout`→`layout`；去除 `or` 兜底；统一 formats 词汇 | platform: submodule_mapping/submodule_models; engine: manifest_builder |
| **P4 统一参数模型** | `use_native_renderer` 单一命名；histogram_ 前缀；WorkflowRequest 与 SubmoduleRequest 参数对齐 | platform: requests/models, submodule_models; plugin: _core |
| **P5 onestop 编排子模块** | planner 复用子模块语义，不再改写 engine_workflow；`handlers/pairwise.py` 重写逻辑下沉为计划层 target-gene 分支 | platform: planning/planner, execution/handlers/pairwise |
| **P6 资产与文档重生** | 重命名/删除资产目录、build featureMap、entry 文件、tests、ARCHITECTURE/PARAMETER_MAPPING/README | integrations/haiant_plugin/* |

> P1→P2→...→P6 顺序保证每步可独立通过测试；P1 与 P2 是核心解耦，P3-P5 收口一致性，P6 对齐对外资产。

---

## 5. 验收标准

1. 每阶段结束：`ruff check` + `ruff format --check` + `pytest`（platform / engines/jcvi / integrations/haiant_plugin）全绿。
2. 渲染模块在缺上游产物时**报明确错误**，不再静默重算 pairwise。
3. `jcvi.pairwise` 单模块可产出 anchors/blocks/simple，`emit_ortholog=true` 时附带 ortholog catalog。
4. 不存在 `catalog_ortholog`、`graphics_*`、`local_synteny_multi` 等旧 module_id / 旧 workflow 字符串 / 旧端口别名。
5. onestop 全局/多物种路径产物与重整前一致（回归对比）。
6. 文档（ARCHITECTURE.md / PARAMETER_MAPPING.md / 各 README / 子模块手册）与新分层一致。

---

## 6. 风险与回滚

- **不可逆性**：干净切换会删除旧资产目录与旧 module_id。回滚依赖 git 历史（本方案各阶段独立提交，可按阶段 revert）。
- **onestop 回归**：P5 改动编排路径，需以重整前的 onestop 产物为基准做回归对比，确认全局核型总图 / 多物种局部总图字节级或语义级一致。
- **engine 别名移除**：P1/P2 移除旧 workflow 字符串后，任何外部缓存的旧 manifest 将失效——符合干净切换预期，但需在交付报告中显式提示。

*本文件随重整推进可修订；每阶段完成后更新对应行的状态。*
