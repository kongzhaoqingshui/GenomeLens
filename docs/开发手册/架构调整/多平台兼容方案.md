# 多平台兼容方案

## 文档定位

这份文档只讨论跨平台运行与发布工程，不重复描述平台长期愿景，也不代替整体架构目标。它回答的问题是：

- 共享核心如何在 Windows、macOS、Linux 上运行
- 平台差异应该收口到哪里
- 工具链、打包、发布和插件边界应如何定义

配套文档如下：

- `最终架构目标.md`：说明长期平台形态
- `温和架构过渡.md`：说明当前如何平滑演进

## 核心结论

GenomeLens 的多平台策略应始终坚持：

- 一套共享核心
- 多平台薄外壳
- 平台专属集成允许存在

也就是说，跨平台的主体是 `Platform Core(平台核心)` 与标准化引擎接口，不是所有外围交付件都必须同步兼容。

## 共享核心与薄外壳

### 共享核心

共享核心是三平台共同复用的部分，至少包括：

- CLI 协议
- 配置协议
- 任务模型
- `manifest(清单)` / `summary(摘要)` 协议
- 输入校验
- 预处理
- 工作流编排
- 结果资产模型

这部分原则上只维护一份代码，不应按平台分叉。

### 薄外壳

薄外壳负责回答“这个平台上怎么跑起来”，而不定义分析语义。它主要处理：

- 可执行文件候选名
- 外部工具链定位
- 外部工具链安装策略
- 打包产物结构
- 平台发布方式
- 平台 CI

这部分可以按平台分别实现，但要通过稳定接口暴露给核心层。

## 平台差异应该收口到哪里

当前和未来都应尽量把平台差异限制在以下区域：

- `platform/src/genomelens/toolchain/runtime/`
- `platform/src/genomelens/toolchain/blast/`
- `platform/src/genomelens/toolchain/imagemagick/`
- `platform/packaging/`
- `engines/jcvi/packaging/`
- 平台专属插件目录

核心工作流与任务模型不应继续直接承担这些差异。

## 可执行文件命名策略

后续代码应默认使用“候选名集合”而不是写死具体文件名。建议统一走类似以下模型：

- `blastn`
- `makeblastdb`
- `magick`
- `jcvi-genomelens`
- `GenomeLens`

平台适配层再负责把它们扩展为：

- Windows：带 `.exe` 的候选名
- macOS / Linux：无扩展名候选

核心层不应继续自己拼接 `.exe`。

## 工具链定位策略

外部工具链定位建议保持统一优先级：

1. CLI 显式参数
2. 配置文件
3. 环境变量
4. 系统 `PATH`
5. 打包资源
6. 本地 `toolchains/`

所有平台都沿用这套顺序，但最终命中的路径来源可以不同。

这意味着：

- Windows 可以更多依赖随包资源和本地缓存
- macOS 更适合优先系统安装或 `brew` / `conda`
- Linux 更适合优先系统安装或 `conda`

## 工具链安装策略

三平台不需要强行共享一种安装方式。建议按平台分别定义：

### Windows

- 允许自动下载并缓存 BLAST+
- 可选缓存 ImageMagick
- 打包时可根据交付等级决定是否内置工具链

### macOS

- 优先系统安装
- 优先支持 `brew` 或 `conda`
- 若当前不支持自动下载，必须返回清晰提示

### Linux

- 优先系统安装
- 优先支持发行版包管理器或 `conda`
- 若当前不支持自动下载，必须返回清晰提示

关键原则不是“三平台同构”，而是“三平台都能被统一接口解释”。

## Cython 与平台策略

JCVI 中的 Cython 扩展会影响跨平台的构建和性能，但不应成为平台化第一阶段的硬门槛。

推荐原则如下：

- 主链路尽量允许无 Cython 扩展时仍可运行
- Cython 应被视为引擎层的核心性能能力，而不是平台核心能力
- 平台打包时可以区分 `core runtime(核心运行时)` 与 `accelerated runtime(加速运行时)`
- Windows 作为当前 `accelerated-first` 的正式平台
- macOS 与 Linux 早期允许先提供 `core runtime`，后续再补正式加速版

这样三平台先追求“可运行”，后追求“同样快”。

更明确地说，当前建议的支持等级如下：

- Windows：官方正式版默认携带或优先验证加速引擎
- macOS：先保证功能与协议正确，再逐步补齐加速构建
- Linux：先保证 CLI / 引擎可运行，再视需求补加速交付

平台核心不应直接理解 `.pyd`、`.so` 或具体编译扩展状态；这些都应由引擎通过 `probe(探测)` 与 `summary(摘要)` 主动声明。

## 打包与发布策略

发布层允许按平台分别设计，但不应改写共享核心。

### 当前建议

- Windows：继续现有发布线
- macOS：优先规划源码运行或轻打包，再逐步补正式桌面交付
- Linux：优先 CLI 交付，必要时再补图形层

### 长期建议

- Tauri GUI 作为 Windows / macOS 的桌面外壳
- Linux 先保留 CLI / API 路线
- 打包脚本按平台分目录维护，不把多平台逻辑硬塞进单个 Windows 脚本

## CI 策略

持续集成建议分层处理：

### 必跑核心 CI

- 共享核心测试
- 标准协议测试
- 当前正式引擎集成测试

### 平台 CI

- Windows：最完整交付验证
- macOS：源码运行与工具链定位验证
- Linux：源码运行与 CLI 验证

### 平台专属 CI

- Windows-only 插件测试
- 平台专属打包 smoke

核心 CI 用来保证平台语义一致，平台 CI 用来保证外壳可运行。

## 插件与集成边界

多平台兼容并不要求所有插件都同步兼容。推荐边界如下：

- 平台核心：追求三平台共享
- 引擎层：按平台构建，但协议统一
- 宿主插件：允许平台专属

以当前项目为例：

- `HAIant plugin` 可以继续保持 Windows-only
- 这不会影响未来 `Platform Core + Engine` 的三平台目标

## 不建议的做法

以下做法会显著增加跨平台维护成本：

- 为 Windows、macOS、Linux 分别维护三套核心逻辑
- 在核心工作流里到处写 `if sys.platform == ...`
- 让打包逻辑侵入任务语义和数据协议
- 让平台专属插件反向决定平台核心设计

## 当前阶段执行重点

在当前阶段，最值得优先执行的是：

1. 停止在新增核心代码中继续扩散平台专属细节
2. 收口 `.exe`、路径、工具链命名和定位逻辑
3. 把安装器策略显式做成平台分发模型
4. 让文档与发布口径明确区分“共享核心”和“平台外壳”

## 验收标准

如果这份方案被正确执行，后续应逐步达到以下状态：

- 核心任务语义在三平台保持一致
- 平台差异主要集中在 runtime 和 packaging 层
- 新增平台时，主要补外围外壳，而不是重写核心逻辑
- 平台专属插件不会污染平台核心

## 结论

GenomeLens 的多平台兼容，真正要兼容的不是“每一个交付件都长得一样”，而是：

- 同一套平台核心语义
- 同一套任务与结果协议
- 同一套引擎接入边界

平台之间允许存在不同的工具链策略、打包方式和插件生态，但这些差异必须被限制在薄外壳层，而不是渗入核心。
