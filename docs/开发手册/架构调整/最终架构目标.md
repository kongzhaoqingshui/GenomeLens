# 最终架构目标

## 文档目的

这份文档定义 GenomeLens 的长期目标架构。该目标同时考虑以下几个方向：

- 从当前以 JCVI 为核心的分析工具，演进为比较基因组学平台。
- 从当前 Windows-first 的实现，演进为共享核心的多平台系统。
- 在用户侧提供基于 Tauri 的桌面 GUI(Graphical User Interface，图形界面) 交付层，用于 Windows 和 macOS。
- 在平台之上逐步接入深度学习能力与 `AI Agent(智能代理)`。

这是一份目标文档，不代表当前版本已经实现其中全部能力。

## 总体目标

GenomeLens 的最终形态不是“一个 JCVI 桌面壳子”，而是“一个以统一任务协议、统一结果资产、可插拔分析引擎和智能助手为核心的比较基因组学平台”。

长期目标可概括为：

- 一套共享的 `Platform Core(平台核心)`。
- 多个可替换、可并列接入的 `Engine(分析引擎)`。
- Windows 和 macOS 共用一套 Tauri 桌面 GUI。
- Linux 至少保留 CLI(API) 可运行能力，后续可按需求补 UI。
- `AI Agent` 和深度学习能力作为平台服务层，而不是写死在某一个引擎里。

## 目标分层

### 1. Platform Core(平台核心)

平台核心是全系统的中心，负责所有平台共享的核心语义：

- 项目、数据集、物种集、分析任务模型。
- `species[]` 输入模型，而不是长期停留在 `query/subject`。
- 任务图、依赖关系、运行状态、审计日志。
- 配置、参数模板、资源需求描述。
- `manifest(清单)`、`result(结果)`、`summary(摘要)` 协议。
- `artifact(结果资产)` 元数据管理。
- `capability registry(能力注册表)`。
- Agent 可调用的任务接口。

这一层应由 `platform/` 逐步演进而来，保持单一核心，不按平台分叉。

#### 当前 V3 平台任务协议

平台核心当前收束为两类公开请求协议：

```text
WorkflowRequest(synteny) -> ExecutionPlan -> PlanOptimizer -> engine manifest v3
SubmoduleRequest(<module_id>) -> ExecutionPlan -> PlanOptimizer -> engine manifest v3
```

对应的职责边界固定为：

```text
GUI / CLI / Plugin / Agent
  -> WorkflowRequest 或 SubmoduleRequest

Platform
  -> 校验、规划、预处理、执行 DAG、归档、summary

JCVI Engine
  -> 具体 JCVI 命令、manifest、绘图、engine summary
```

这意味着 GUI、CLI、插件和 Agent 只能提交用户级任务意图，不直接拼 engine manifest；platform 只解释任务、编排执行和治理产物，不重写具体绘图或 JCVI 命令实现；JCVI engine 只处理标准化 manifest 内的实际计算与图形输出，不反向持有平台业务状态。

- `WorkflowRequest` 仅用于一站式 `synteny`，面向 CLI、GUI、插件和 Agent，使用 `species[]`、`reference_index`、`parameters`、`output`、`runtime` 表达用户意图。
- `SubmoduleRequest` 用于所有原子子模块，使用 `module_id`、`inputs`、`parameters`、`output`、`runtime` 表达用户意图。
- `ExecutionPlan` 是平台内部原始执行计划，由 `WorkflowPlanner` 展开 pairwise、多物种 all-vs-all、reference-vs-targets 等执行 DAG；`SubmoduleDispatcher` 把子模块请求包装为单步 plan。
- `PlanOptimizer` 位于 planning 与 execution 之间，负责去重预处理、共享 toolchain/probe、注入中间产物复用等无损优化。
- `PlanExecutor` 只执行优化后的 plan step，不再在 runner 中重复维护配对策略。
- engine manifest 当前为 `schema_version=3`，顶层使用 `workflow`、`task`、`inputs`、`parameters`、`toolchain`、`expected_outputs`、`meta`。
- `query/subject` 只允许作为 engine 内部局部概念存在，不再作为平台公开入口或 manifest 顶层字段。
- 结果摘要使用 `RunSummary` schema v3，扩展信息进入 `extensions`，pairwise 子任务进入 `child_runs`，artifact 使用可追踪的 `ArtifactRecord`。

### 2. Engine Layer(引擎层)

JCVI 只是第一个正式引擎，不应继续等同于平台本体。长期目标下，GenomeLens 应允许并列接入多个引擎，例如：

- `jcvi-engine`
- `mcscanx-engine`
- `syri-engine`
- `pangenome-engine`
- `orthology-engine`
- `dl-model-engine`

每个引擎都通过统一接口被平台调度：

- `probe`
- `run`
- `manifest schema`
- `result schema`
- `artifact declaration`

平台核心不直接依赖某个引擎的私有命令格式。

对于带 Cython 的高性能引擎，长期建议采用以下支持口径：

- Windows 优先提供 `accelerated runtime(加速运行时)`
- 其他平台早期允许先提供 `core runtime(核心运行时)`
- 引擎必须能通过 `probe` 和运行摘要声明当前运行模式与已加载扩展

这使得高性能能力可以成为引擎竞争力的一部分，同时不污染平台核心。

### 3. Workflow Layer(工作流层)

工作流层表达的是“用户要完成什么分析任务”，而不是“底层调用哪个程序”。长期应支持的工作流示例包括：

- 双物种共线性分析
- 多物种共线性分析
- 正交基因分析
- 泛基因组相关流程
- 结构变异比较
- 图件生成与美化
- 深度学习预测流程
- 自动化报告流程

工作流层由平台核心编排，再根据能力注册表选择合适引擎执行。

### 4. Artifact Layer(结果资产层)

长期目标下，GenomeLens 的输出不能只是“某个目录下的一堆文件”，而应是可追踪的结果资产体系。每个产物都应至少具备以下元数据：

- `artifact_id`
- `artifact_type`
- `produced_by`
- `input_refs`
- `owner_task`
- `format`
- `preview`
- `provenance(来源追踪)`

这层会直接决定平台未来的可复用性、可审计性和 Agent 解释能力。

### 5. Agent Layer(智能代理层)

`AI Agent` 不应直接绕过平台执行底层脚本，而应通过平台任务接口工作。长期可承担的职责包括：

- 工作流规划
- 参数建议
- 输入质量检查
- 错误诊断
- 图件解释
- 结果对比
- 文献辅助
- 自动生成报告草稿

`AI Agent` 是平台服务层，不是某个具体引擎的附属模块。

### 6. ML Scoring Layer(机器学习评分层)

长期目标下，GenomeLens 还应允许在基础比较基因组结果之上叠加机器学习评分能力。这一层适合承载以下类型的能力：

- 候选同源关系可信度评分
- 启动子差异风险评分
- 区段综合评分
- 其他基于比较基因组结果、序列特征和上下文特征的模型评分

这一层的定位应明确如下：

- 单个评分模型更适合做成独立 `Engine(引擎)` 或模型服务。
- 多个评分结果的综合排序更适合由平台服务层或工作流层完成。
- 评分结果必须进入统一 `artifact(结果资产)` 与 `summary(摘要)` 体系，而不是只存在于界面临时计算中。

可预期的引擎示例包括：

- `orthology-scoring-engine`
- `promoter-risk-engine`
- `region-scoring-engine`
- `ranking-aggregation-service`

这层的存在意义，是让 GenomeLens 不只提供“工具结果”，还提供“面向决策的定量解释”。

### 7. GUI Layer(Tauri 图形界面层)

桌面 GUI 长期建议采用 Tauri，并把它明确定位为 Win/Mac 的用户交互外壳，而不是新的业务核心。

Tauri 层负责：

- 项目浏览与任务创建
- 参数表单与模板选择
- 运行进度与日志展示
- 结果资产浏览
- 图件预览
- Agent 对话入口
- 评分结果、排序结果与解释字段展示
- 本地设置和运行环境诊断

Tauri 层不负责：

- 直接实现分析算法
- 持有平台核心业务规则
- 单独维护一套与 CLI 不一致的任务协议

Tauri 应通过本地 API、命令桥接或 sidecar(伴随进程) 与平台核心通信，而不是把核心逻辑迁入前端。

## 多平台目标

长期目标下，各平台角色建议明确如下：

- Windows：CLI + Engine + Tauri GUI + 平台专属插件
- macOS：CLI + Engine + Tauri GUI
- Linux：CLI + Engine，GUI 后续按需求决定

这意味着“多平台兼容”的主体是平台核心和引擎层，不是所有集成件都必须同步跨平台。

## Tauri 与核心的关系

长期最稳的模式是：

- `Tauri` 做桌面宿主和图形交互。
- `Platform Core` 做任务编排、状态管理和协议解释。
- `Engine` 做真实计算。

推荐通信边界：

- GUI 只提交任务、查询状态、读取结果。
- 核心只暴露稳定任务接口，不暴露临时脚本细节。
- 引擎只接受标准化 `manifest(清单)` 并输出标准化 `summary(摘要)`。

不要让 Tauri 直接去拼 BLAST、JCVI 或 Python 子命令细节。

## 平台边界

长期应明确以下边界：

- 平台核心可以三平台共享。
- 引擎可以按平台分别打包，但必须遵循统一协议。
- Tauri GUI 当前只服务 Windows 和 macOS。
- 宿主插件允许平台专属，不必绑架平台核心。
- 深度学习模型属于引擎能力，不应写进 GUI 层。
- 机器学习评分模型优先视为引擎能力或模型服务，不应直接塞进 GUI 或平台脚本。
- 候选优先级排序属于平台工作流或服务层，不宜绑定到单一底层引擎。
- Agent 通过平台接口调度，不直接控制平台专属脚本。

## 架构原则

最终架构应始终坚持以下原则：

- 平台核心单一。
- 引擎能力插件化。
- 任务协议稳定。
- 结果资产可追踪。
- GUI 只是外壳，不是第二核心。
- Agent 是调度者，不是绕过平台的脚本执行者。

## 结论

GenomeLens 的最终目标，不应只是“把 JCVI 做成更完整的桌面程序”，而应是：

一个以共享平台核心为中心，允许多引擎接入、支持多平台运行、提供 Tauri 桌面 GUI、并可承载深度学习与 AI Agent 的比较基因组学平台。
