# 更新计划文档维护规则

## 适用范围

当用户提到以下任意场景时使用本技能：

- “更新计划/计划更新的内容.md”
- “计划更新”
- “归档历史”
- “给出见解并写成内容方案”
- 对版本规划、需求编号、开发顺序、协作分工提出建议

## 工作流程

### 1. 读取上下文

在给出方案前，先读取：

- `docs/更新计划/计划更新的内容.md`（当前计划）
- `docs/历史/计划更新的内容.md`（历史归档）
- `docs/更新计划/更新日志.md`（已发布内容）

### 2. 归档已完成项

- 若当前计划中存在已发布/已完成的需求或版本，将其迁移到历史文档的“已发布版本归档”中。
- 历史文档保留：版本号、发布日期、原始编号、关键交付物、完成状态。
- 当前计划只保留未实现或部分实现的需求；已完成的编号说明可简写为“已完成并归档”。

### 3. 根据用户方向生成内容方案

对用户在当前计划中指出的方向，依次给出：

- **见解**：可行性、与现有架构的契合度、依赖是否就绪、风险点、建议归属的版本。
- **具体条目**：
  - 编号（沿用现有编号；新增时按顺序递增，不跳号）
  - 需求简述
  - 类型（功能 / 基础 / 体验 / 集成 / 平台 / 发布）
  - 优先级（高 / 中 / 低）
  - 依赖（若依赖已完成，注明“已就绪”）
  - 版本归属（v0.9.x / v1.0.0 / v1.1.0 / v1.2.0 等）
  - 验收标准或关键任务列表（使用复选框）
- **联动更新**：同步修改以下章节，使四节互相引用、无脱节：
  - `## 编号说明`
  - `## 版本分布`
  - `## 建议开发顺序`
  - `## 多人协作分工`
  - `## 代码审查重点`
  - `## 风险与注意事项`

### 4. 一致性要求

- 编号说明、版本分布、开发顺序、协作分工四节必须互相引用。
- 新增需求必须标注依赖；若依赖已完成，注明“已就绪”。
- 已完成项不留在当前计划的版本分布中，只保留未完成任务。
- 同一需求不要同时出现在“当前计划”和“历史归档”的活跃部分。

### 5. 语言与格式

- 全部使用中文。
- 使用 Markdown 复选框 `-[ ]` / `-[x]` 表示完成状态。
- 版本标题格式：`### vX.Y.Z — 主题`。
- 文件变更后，若涉及发布，同步更新 `docs/更新计划/更新日志.md` 与 `platform/src/genomelens/_version.py`。

## 示例

用户说：“v1.0.0 前要加一个性能回归测试框架”。

应给出的见解与方案：

- 见解：属于基础设施，建议在 v0.9.x 末尾或 v1.0.0 发布前落地，不阻塞功能开发；依赖 CI 与现有测试框架，已就绪。
- 方案：
  - 在编号说明新增条目 10：性能回归测试框架 / 基础 / 高 / 依赖：pytest、CI（已就绪）。
  - 在 v0.9.5 或 v1.0.0 增加子任务：定义基准数据集、添加回归测试工作流、对比耗时/内存阈值。
  - 更新开发顺序：在“v1.0.0 发布准备”之前插入该任务。
  - 更新审查重点：新增“性能回归测试是否覆盖主要 workflow，阈值是否合理”。
